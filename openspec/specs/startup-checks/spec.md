# Capability: startup-checks

## Purpose

Проверки при старте приложения: связность с PostgreSQL, валидность Telegram-токена, порядок выполнения startup-шагов.

## Requirements

### Requirement: Проверка связности с PostgreSQL при старте

При запуске приложения, после применения миграций, система ДОЛЖНА выполнить тестовый запрос к PostgreSQL (`SELECT 1`). При неудаче — завершить процесс с ненулевым кодом и описательным сообщением.

#### Scenario: PostgreSQL доступен
- **WHEN** приложение запускается и PostgreSQL отвечает на `SELECT 1`
- **THEN** запуск продолжается

#### Scenario: PostgreSQL недоступен
- **WHEN** приложение запускается, но PostgreSQL не отвечает (connection refused, timeout)
- **THEN** приложение логирует ошибку «Не удалось подключиться к PostgreSQL: {error}» и завершается с кодом 1

### Requirement: Проверка Telegram-токена при старте

При запуске приложения система ДОЛЖНА вызвать `bot.get_me()` для проверки валидности токена. При невалидном токене — завершить процесс с ненулевым кодом.

#### Scenario: Токен валидный
- **WHEN** приложение запускается и `bot.get_me()` возвращает информацию о боте
- **THEN** запуск продолжается, логируется имя бота

#### Scenario: Токен невалидный
- **WHEN** приложение запускается, но `bot.get_me()` возвращает ошибку аутентификации
- **THEN** приложение логирует ошибку «Невалидный Telegram-токен» и завершается с кодом 1

### Requirement: Порядок проверок при старте

Проверки ДОЛЖНЫ выполняться в определённом порядке: (1) env-переменные (pydantic-settings, уже есть), (2) PostgreSQL connectivity, (3) Alembic миграции, (4) Telegram token. Если шаг N завершился ошибкой, шаги N+1... не выполняются.

#### Scenario: Все проверки прошли
- **WHEN** все env-переменные заданы, PostgreSQL доступен, миграции применены, токен валидный
- **THEN** запускается APScheduler и long-polling

#### Scenario: Ошибка на шаге PostgreSQL
- **WHEN** env-переменные валидны, но PostgreSQL недоступен
- **THEN** миграции не запускаются, токен не проверяется, приложение завершается
