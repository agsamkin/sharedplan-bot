# Capability: event-creation

## Purpose

Создание событий через Telegram-бота: перехват текстовых сообщений, AI-парсинг естественного языка через LLM, выбор пространства, карточка подтверждения, сохранение в БД, уведомление участников.

## Requirements

### Requirement: Перехват текстовых сообщений для создания событий

Обработчик в `bot/handlers/event.py` ДОЛЖЕН перехватывать все текстовые сообщения, не являющиеся командами (через фильтр `NotCommandFilter`), и инициировать флоу создания события.

#### Scenario: Пользователь отправляет обычный текст
- **WHEN** пользователь отправляет текстовое сообщение, не начинающееся с `/`
- **THEN** запускается флоу создания события: парсинг текста -> выбор пространства -> карточка подтверждения

#### Scenario: Пользователь отправляет команду
- **WHEN** пользователь отправляет сообщение, начинающееся с `/`
- **THEN** сообщение НЕ обрабатывается хендлером событий, а маршрутизируется к соответствующему обработчику команды

### Requirement: Парсинг структурированного текста (заглушка)

Заглушка `parse_event_text()` ЗАМЕНЯЕТСЯ на вызов `llm_parser.parse_event()`. Хендлер `bot/handlers/event.py` ДОЛЖЕН передавать текст пользователя в LLM-парсер вместо ручного парсинга по формату.

При ошибке парсинга хендлер ДОЛЖЕН отображать понятное сообщение в зависимости от типа ошибки, вместо подсказки о формате ввода.

#### Scenario: Пользователь пишет естественным языком
- **WHEN** пользователь отправляет «Ужин с родителями завтра в 19:00»
- **THEN** текст передаётся в `llm_parser.parse_event()`, результат отображается в карточке подтверждения

#### Scenario: LLM не смог распознать событие
- **WHEN** `llm_parser.parse_event()` выбрасывает `ParseError` с типом `invalid_json`
- **THEN** бот отвечает: «Не удалось распознать событие. Попробуй переформулировать.»

#### Scenario: Таймаут или ошибка сети
- **WHEN** `llm_parser.parse_event()` выбрасывает `ParseError` с типом `timeout` или `network`
- **THEN** бот отвечает: «Ошибка соединения, попробуй позже.»

#### Scenario: Rate limit исчерпан
- **WHEN** `llm_parser.parse_event()` выбрасывает `ParseError` с типом `rate_limit`
- **THEN** бот отвечает: «Сервис временно недоступен, попробуй через минуту.»

#### Scenario: Индикатор набора текста
- **WHEN** пользователь отправляет текстовое сообщение для создания события
- **THEN** бот отправляет `chat_action: typing` перед вызовом LLM

### Requirement: Автоматический выбор пространства

Если пользователь состоит в одном пространстве, оно ДОЛЖНО выбираться автоматически. Если пользователь состоит в нескольких — ДОЛЖНА отображаться inline-клавиатура выбора.

#### Scenario: Пользователь в одном пространстве
- **WHEN** пользователь отправляет текст для создания события и состоит в одном пространстве
- **THEN** пространство выбирается автоматически, сразу отображается карточка подтверждения

#### Scenario: Пользователь в нескольких пространствах
- **WHEN** пользователь отправляет текст для создания события и состоит в нескольких пространствах
- **THEN** отображается inline-клавиатура с названиями пространств, парсированное событие сохраняется в FSM-данных

#### Scenario: Пользователь не состоит ни в одном пространстве
- **WHEN** пользователь отправляет текст, но не состоит ни в одном пространстве
- **THEN** бот отвечает: «Сначала создай пространство через /newspace»

### Requirement: Карточка подтверждения события

После парсинга и выбора пространства бот ДОЛЖЕН отправить карточку подтверждения с данными события и тремя inline-кнопками.

#### Scenario: Событие с временем
- **WHEN** парсинг вернул событие с датой и временем
- **THEN** бот отправляет карточку формата: `📌 Событие:\n📝 {title}\n📅 {date}\n⏰ {time}\n\nОпубликовать?` с кнопками `[✅ Да]` `[✏️ Изменить]` `[❌ Отмена]`

#### Scenario: Событие без времени
- **WHEN** парсинг вернул событие без времени (time=None)
- **THEN** карточка отображается без строки ⏰

### Requirement: Подтверждение и сохранение события

При нажатии кнопки «✅ Да» событие ДОЛЖНО сохраняться в таблицу `events` и уведомления ДОЛЖНЫ отправляться участникам пространства.

#### Scenario: Пользователь подтверждает событие
- **WHEN** пользователь нажимает кнопку `✅ Да` (callback `event_confirm`)
- **THEN** создаётся запись в `events` с полями `space_id`, `title`, `event_date`, `event_time`, `created_by`, `raw_input`, отправляются уведомления участникам, карточка обновляется сообщением об успехе

#### Scenario: Таблица scheduled_reminders не заполняется
- **WHEN** событие подтверждается и сохраняется
- **THEN** записи в `scheduled_reminders` НЕ создаются (функциональность Фазы 6)

### Requirement: Отмена создания события

При нажатии кнопки «❌ Отмена» создание события ДОЛЖНО отменяться.

#### Scenario: Пользователь отменяет создание
- **WHEN** пользователь нажимает кнопку `❌ Отмена` (callback `event_cancel`)
- **THEN** карточка обновляется сообщением «Отменено», FSM-данные очищаются, запись в БД не создаётся

### Requirement: Редактирование черновика события

При нажатии кнопки «Изменить» бот ДОЛЖЕН попросить пользователя отправить исправленный вариант в свободной форме (без указания формата).

Повторный ввод ДОЛЖЕН парситься через `llm_parser.parse_event()`.

#### Scenario: Пользователь нажимает «Изменить»
- **WHEN** пользователь нажимает кнопку `✏️ Изменить` (callback `event_edit`)
- **THEN** бот отправляет сообщение «Отправь исправленный вариант», FSM переходит в состояние ожидания нового ввода

#### Scenario: Пользователь отправляет исправленный вариант
- **WHEN** пользователь в состоянии редактирования отправляет новый текст
- **THEN** текст парсится через `llm_parser.parse_event()` и отображается обновлённая карточка подтверждения

#### Scenario: Ошибка парсинга при редактировании
- **WHEN** LLM не смог распознать исправленный текст
- **THEN** пользователь получает сообщение об ошибке и остаётся в состоянии редактирования

### Requirement: Уведомление участников пространства

После сохранения события все участники пространства (кроме создателя) ДОЛЖНЫ получить уведомление.

#### Scenario: Уведомление при публикации
- **WHEN** событие подтверждено и сохранено в пространстве с 3 участниками
- **THEN** 2 участника (кроме создателя) получают сообщение формата: `📅 Новое событие в «{space_name}»!\n\n📝 {title}\n📅 {date}\n⏰ {time}\n👤 Добавил: {creator_name}`

#### Scenario: Событие без времени — уведомление без ⏰
- **WHEN** событие без времени подтверждено
- **THEN** уведомление отправляется без строки ⏰

#### Scenario: Ошибка отправки одному участнику
- **WHEN** отправка уведомления одному участнику не удалась (пользователь заблокировал бота)
- **THEN** ошибка логируется, уведомления остальным участникам отправляются нормально

#### Scenario: Создатель единственный участник
- **WHEN** событие подтверждено, но создатель — единственный участник пространства
- **THEN** уведомления никому не отправляются (нет других участников)

### Requirement: Сервисный слой event_service.py

Бизнес-логика событий ДОЛЖНА быть в `services/event_service.py`, отделённая от Telegram-специфичного кода.

#### Scenario: Создание события через сервис
- **WHEN** хендлер вызывает `create_event(session, space_id, title, event_date, event_time, created_by, raw_input)`
- **THEN** сервис создаёт запись в таблице `events` и возвращает объект `Event`

#### Scenario: Получение участников для уведомления
- **WHEN** хендлер запрашивает список участников для уведомления
- **THEN** сервис возвращает список участников пространства, исключая создателя события
