# Capability: miniapp-space-detail

## Purpose

Экраны детального просмотра, редактирования и управления пространством в Mini App: Space Detail, Space Edit, Members, а также переработанный экран EventDetail.

## Requirements

### Requirement: Экран Space Detail
Система SHALL отображать экран детального просмотра пространства по маршруту `/spaces/:id`. Экран SHALL содержать: панель action-кнопок сверху, секцию со списком событий. Заголовок Header SHALL показывать имя пространства. Секция событий SHALL использовать `total_count` из API для отображения общего количества событий в заголовке (формат: "События · {total_count}"). Секция событий SHALL содержать кнопку «Добавить» в заголовке для навигации к экрану создания события.

#### Scenario: Просмотр пространства с событиями
- **WHEN** пользователь переходит на экран пространства, API возвращает `{ events: [...5], total_count: 5 }`
- **THEN** отображается панель действий и секция "События · 5" со списком из 5 событий, кнопка «Добавить» в заголовке секции

#### Scenario: Пустое пространство
- **WHEN** пространство не содержит событий, API возвращает `{ events: [], total_count: 0 }`
- **THEN** отображается секция "События · 0" с сообщением "Нет предстоящих событий", кнопка «Добавить» в заголовке секции

#### Scenario: Больше событий чем отображается
- **WHEN** API возвращает `{ events: [...50], total_count: 75 }`
- **THEN** секция показывает "События · 75", список содержит 50 событий

### Requirement: Кнопка «Добавить» в секции событий
Система SHALL отображать кнопку «Добавить» справа в заголовке секции событий. Кнопка SHALL иметь иконку "+" и текст «Добавить», стиль: background #378ADD14, color #378ADD, borderRadius 8px, fontSize 13, fontWeight 600. При нажатии SHALL навигировать к `/spaces/:id/events/new`.

#### Scenario: Нажатие кнопки «Добавить»
- **WHEN** пользователь нажимает кнопку «Добавить» в секции событий
- **THEN** происходит навигация на `/spaces/:id/events/new`

#### Scenario: Кнопка видна всем участникам
- **WHEN** пользователь является участником пространства (любая роль)
- **THEN** кнопка «Добавить» отображается в заголовке секции событий

### Requirement: Панель action-кнопок
Система SHALL отображать горизонтальную панель action-кнопок с белым фоном, padding 16px, gap 12px, выровненных по центру. Для администратора SHALL отображаться 4 кнопки: "Изменить" (blue, IconEdit), "Участники" (blue, IconPeople), "Ссылка" (purple, IconLink), "Удалить" (red, IconTrash). Для обычного участника SHALL отображаться только "Участники" и "Ссылка".

#### Scenario: Администратор видит все действия
- **WHEN** пользователь является администратором пространства
- **THEN** отображаются все 4 action-кнопки

#### Scenario: Участник видит ограниченные действия
- **WHEN** пользователь является обычным участником пространства
- **THEN** отображаются только кнопки "Участники" и "Ссылка"

### Requirement: Список событий с DateBadge
Система SHALL отображать события пространства в виде списка ListItem. Каждое событие SHALL иметь: DateBadge слева (число + месяц), title -- название, subtitle -- "относительная_дата * время * автор", ChevronRight справа. События SHALL быть разделены Divider.

#### Scenario: Событие с временем
- **WHEN** событие "Ужин с родителями" на 29 марта в 19:00, автор Иван
- **THEN** ListItem показывает DateBadge "29 МАР", title "Ужин с родителями", subtitle "Завтра * 19:00 * Иван"

#### Scenario: Событие без времени
- **WHEN** событие "День рождения Ани" на 5 апреля без времени, автор Мария
- **THEN** subtitle не содержит время: "5 апр * Мария"

### Requirement: Переход к редактированию события
Система SHALL при нажатии на событие в списке навигировать к экрану EventDetail (`/events/:id`), передавая данные события.

#### Scenario: Нажатие на событие
- **WHEN** пользователь нажимает на строку события в списке
- **THEN** происходит навигация на `/events/:eventId`

### Requirement: Действие "Ссылка" копирует invite-ссылку
Система SHALL при нажатии кнопки "Ссылка" копировать invite-ссылку пространства в буфер обмена и показывать Toast "Ссылка скопирована".

#### Scenario: Копирование ссылки
- **WHEN** пользователь нажимает кнопку "Ссылка" на панели действий
- **THEN** invite-ссылка `https://t.me/{bot}?start=join_{code}` копируется в clipboard и появляется Toast

### Requirement: Действие "Удалить" с инлайн-подтверждением
Система SHALL при нажатии кнопки "Удалить" отображать инлайн-блок подтверждения. При подтверждении SHALL вызываться API `DELETE /api/spaces/:id`, после успеха -- навигация на `/` и Toast "Пространство удалено".

#### Scenario: Удаление пространства
- **WHEN** администратор подтверждает удаление пространства "Семья"
- **THEN** API удаляет пространство, пользователь возвращается на главный экран, появляется Toast "Пространство удалено"

### Requirement: Действие "Изменить" навигирует к Space Edit
Система SHALL при нажатии кнопки "Изменить" навигировать к экрану `/spaces/:id/edit` для редактирования названия пространства.

#### Scenario: Переход к редактированию
- **WHEN** администратор нажимает кнопку "Изменить"
- **THEN** происходит навигация на экран Space Edit

### Requirement: Экран Space Edit
Система SHALL отображать экран редактирования названия пространства по маршруту `/spaces/:id/edit`. Экран SHALL содержать: Section с заголовком "Название пространства", input с текущим именем (autoFocus), кнопку "Сохранить" (disabled если пустое). При сохранении SHALL вызываться API и показываться Toast "Сохранено".

#### Scenario: Редактирование названия
- **WHEN** администратор меняет название с "Семья" на "Моя семья" и нажимает "Сохранить"
- **THEN** название обновляется, пользователь возвращается назад, появляется Toast "Сохранено"

#### Scenario: Пустое название
- **WHEN** поле названия пустое
- **THEN** кнопка "Сохранить" неактивна (opacity 0.4)

### Requirement: Экран Members
Система SHALL отображать список участников пространства по маршруту `/spaces/:id/members`. Каждый участник SHALL отображаться как ListItem: Avatar слева, имя title, "@username * администратор" subtitle. Администраторы SHALL иметь бейдж "admin" (фон #EEEDFE, текст purple). Для обычных участников администратор SHALL видеть кнопку "Удалить" (красная).

#### Scenario: Список участников
- **WHEN** пространство имеет 4 участника, пользователь -- администратор
- **THEN** отображаются все 4 участника, у не-администраторов -- кнопка "Удалить", у администратора -- бейдж "admin"

#### Scenario: Удаление участника
- **WHEN** администратор нажимает "Удалить" у участника и подтверждает инлайн
- **THEN** API `DELETE /api/spaces/:id/members/:userId` удаляет участника, появляется Toast

### Requirement: Переработанный экран EventDetail

Система SHALL отображать экран просмотра/редактирования события по маршруту `/events/:id`. При загрузке события система SHALL использовать поле `is_owner` из ответа API для определения режима отображения.

**Режим владельца** (`is_owner === true`): экран SHALL содержать Section "Детали события" с редактируемыми полями Название (input), Дата (input type="date"), Время (input type="time"), Section с информацией об авторе (IconPerson + "Создал: имя"), кнопку "Сохранить" (primary blue), кнопку "Удалить событие" (danger outline). Удаление SHALL использовать инлайн-подтверждение. Заголовок Header SHALL быть "Редактирование".

**Режим просмотра** (`is_owner === false`): экран SHALL содержать Section "Детали события" с полями в режиме read-only (атрибут `readOnly` на input-элементах, приглушённый стиль без border), Section с информацией об авторе. Кнопки "Сохранить" и "Удалить событие" SHALL НЕ отображаться. Заголовок Header SHALL быть "Событие".

#### Scenario: Владелец открывает своё событие
- **WHEN** пользователь открывает экран `/events/:id` и API возвращает `is_owner: true`
- **THEN** отображается заголовок "Редактирование", поля редактируемые, кнопки "Сохранить" и "Удалить событие" видны

#### Scenario: Участник открывает чужое событие
- **WHEN** пользователь открывает экран `/events/:id` и API возвращает `is_owner: false`
- **THEN** отображается заголовок "Событие", поля в режиме read-only (readOnly, приглушённый стиль), кнопки "Сохранить" и "Удалить событие" отсутствуют

#### Scenario: Редактирование и сохранение владельцем
- **WHEN** владелец меняет название события и нажимает "Сохранить"
- **THEN** API `PUT /api/events/:id` обновляет событие, навигация назад, Toast "Событие обновлено"

#### Scenario: Удаление события владельцем
- **WHEN** владелец нажимает "Удалить событие" и подтверждает
- **THEN** API `DELETE /api/events/:id` удаляет событие, навигация назад, Toast "Событие удалено"

#### Scenario: Не-владелец не может редактировать через UI
- **WHEN** не-владелец находится на экране события с `is_owner: false`
- **THEN** input-поля имеют атрибут `readOnly`, кнопки управления отсутствуют в DOM, попытка взаимодействия с полями не приводит к изменению значений
