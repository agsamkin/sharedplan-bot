# Capability: bot-access-control

## Purpose

Контроль доступа к боту: только владелец и участники пространств могут использовать бота, остальные получают отказ (invite-only модель).

## Requirements

### Requirement: Middleware контроля доступа

Система ДОЛЖНА включать `AccessControlMiddleware`, зарегистрированный на `dp.update.middleware()` после `DbSessionMiddleware` и перед `UserProfileMiddleware`. Middleware ДОЛЖЕН проверять авторизацию пользователя на каждом входящем update.

#### Scenario: Владелец бота всегда авторизован

- **WHEN** update приходит от пользователя с Telegram ID, совпадающим с `OWNER_TELEGRAM_ID`
- **THEN** update передаётся следующему middleware/handler без дополнительных проверок

#### Scenario: Участник пространства авторизован

- **WHEN** update приходит от пользователя, имеющего хотя бы одну запись в `user_spaces`
- **THEN** update передаётся следующему middleware/handler

#### Scenario: Неавторизованный пользователь заблокирован

- **WHEN** update приходит от пользователя, который не является владельцем и не имеет записей в `user_spaces`
- **AND** update не является командой `/start join_*`
- **THEN** бот отправляет сообщение «Этот бот доступен только по приглашению. Попроси ссылку у владельца.» и прерывает обработку (handler не вызывается)

#### Scenario: Deep link join пропускается для неавторизованных

- **WHEN** update содержит message с текстом, начинающимся с `/start join_`
- **AND** пользователь не авторизован
- **THEN** update передаётся handler-у для обработки присоединения к пространству

#### Scenario: Callback query от неавторизованного пользователя

- **WHEN** callback_query приходит от неавторизованного пользователя
- **THEN** бот отвечает на callback_query текстом «Доступ запрещён» и прерывает обработку

### Requirement: Порядок регистрации middleware

Middleware ДОЛЖНЫ быть зарегистрированы в следующем порядке:
1. `DbSessionMiddleware` — создание async session
2. `AccessControlMiddleware` — проверка доступа
3. `UserProfileMiddleware` — upsert профиля пользователя

`UserProfileMiddleware` НЕ ДОЛЖЕН выполняться для заблокированных пользователей.

#### Scenario: Заблокированный пользователь не создаёт профиль

- **WHEN** неавторизованный пользователь отправляет сообщение боту
- **THEN** запись в таблице `users` НЕ создаётся и НЕ обновляется для этого пользователя

### Requirement: Поведение /start для неавторизованных

Команда `/start` без параметров ДОЛЖНА быть недоступна для неавторизованных пользователей. Middleware ДОЛЖЕН перехватить такой update и показать сообщение о приватности бота.

#### Scenario: Неавторизованный пользователь отправляет /start

- **WHEN** неавторизованный пользователь отправляет `/start` без параметров
- **THEN** бот отвечает: «Этот бот доступен только по приглашению. Попроси ссылку у владельца.»

#### Scenario: Авторизованный пользователь отправляет /start

- **WHEN** авторизованный пользователь (владелец или участник пространства) отправляет `/start`
- **THEN** бот показывает обычное приветственное сообщение

### Requirement: Изоляция ошибок в middleware

Ошибка при проверке доступа (например, сбой БД) НЕ ДОЛЖНА приводить к пропуску неавторизованного пользователя. При ошибке проверки — запрос ДОЛЖЕН быть отклонён (fail-closed).

#### Scenario: Ошибка при проверке user_spaces

- **WHEN** запрос к таблице `user_spaces` завершается ошибкой
- **THEN** update отклоняется, ошибка логируется, пользователь получает сообщение об ошибке
