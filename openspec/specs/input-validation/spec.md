# Capability: input-validation

## Purpose

Валидация пользовательского ввода: длина текста событий, диапазон дат, предупреждения о прошедших датах, длина названий пространств.

## Requirements

### Requirement: Валидация длины текста события

Система ДОЛЖНА проверять длину текстового ввода пользователя перед отправкой в LLM-парсер. Допустимый диапазон: 1–1000 символов. Пустой текст и текст длиннее 1000 символов ДОЛЖНЫ отклоняться с понятным сообщением.

#### Scenario: Текст в допустимом диапазоне
- **WHEN** пользователь отправляет текст длиной 50 символов
- **THEN** текст передаётся в LLM-парсер для создания события

#### Scenario: Пустой текст
- **WHEN** пользователь отправляет пустое сообщение или сообщение из пробелов
- **THEN** бот не вызывает LLM-парсер и не отвечает ошибкой (сообщение игнорируется хендлером событий)

#### Scenario: Текст длиннее 1000 символов
- **WHEN** пользователь отправляет текст длиной 1500 символов
- **THEN** бот отвечает: «Слишком длинное сообщение. Опиши событие короче (до 1000 символов).» и не вызывает LLM-парсер

### Requirement: Предупреждение о дате в прошлом

Если LLM-парсер вернул дату, которая уже прошла, система ДОЛЖНА показать предупреждение с возможностью подтвердить или отменить создание. Событие НЕ ДОЛЖНО создаваться автоматически, если дата в прошлом.

#### Scenario: Дата в прошлом — предупреждение
- **WHEN** LLM-парсер вернул событие с `event_date` < текущей даты
- **THEN** бот показывает карточку: «⚠️ Дата уже прошла ({date}).\n\n📝 {title}\n\nВсё равно создать?» с кнопками `[✅ Всё равно создать]` и `[❌ Отмена]`

#### Scenario: Пользователь подтверждает прошедшую дату
- **WHEN** пользователь нажимает «✅ Всё равно создать» на предупреждении о прошедшей дате
- **THEN** отображается стандартная карточка подтверждения события (как для обычного события)

#### Scenario: Пользователь отменяет прошедшую дату
- **WHEN** пользователь нажимает «❌ Отмена» на предупреждении о прошедшей дате
- **THEN** создание отменяется, сообщение обновляется на «Отменено»

#### Scenario: Дата сегодня — без предупреждения
- **WHEN** LLM-парсер вернул событие с `event_date` = текущей дате
- **THEN** стандартный флоу подтверждения, предупреждение не показывается

### Requirement: Валидация диапазона даты

Система ДОЛЖНА отклонять даты более чем 2 года в будущее. Такие даты обрабатываются как ошибка парсинга.

#### Scenario: Дата в допустимом диапазоне
- **WHEN** LLM-парсер вернул событие с `event_date` через 6 месяцев от текущей даты
- **THEN** стандартный флоу подтверждения

#### Scenario: Дата более 2 лет в будущее
- **WHEN** LLM-парсер вернул событие с `event_date` через 3 года от текущей даты
- **THEN** выбрасывается `ParseError` с типом `invalid_json`, бот отвечает: «Не удалось распознать событие. Попробуй переформулировать.»

### Requirement: Валидация длины названия пространства

Система ДОЛЖНА проверять длину названия пространства при создании: 1–255 символов. Это уже реализовано — требование формализует существующее поведение.

#### Scenario: Название пустое
- **WHEN** пользователь вводит пустое название пространства
- **THEN** бот просит ввести название

#### Scenario: Название длиннее 255 символов
- **WHEN** пользователь вводит название пространства длиной 300 символов
- **THEN** бот отвечает: «Слишком длинное название. Максимум 255 символов.»
