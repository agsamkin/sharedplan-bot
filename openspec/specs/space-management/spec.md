# Capability: space-management

## Purpose

Управление пространствами (spaces): создание, просмотр, удаление пространств, управление участниками (kick), inline-клавиатура выбора пространства, сервисный слой, вспомогательные фильтры.

## Requirements

### Requirement: Создание пространства через /newspace с FSM

Команда `/newspace` ДОЛЖНА запускать FSM-флоу создания пространства. Бот ДОЛЖЕН запросить название, валидировать его (1-255 символов), сгенерировать уникальный invite-код через `secrets.token_urlsafe(6)`, создать записи в `spaces` и `user_spaces` (role='admin') в одной транзакции, и ответить подтверждением с invite-ссылкой.

#### Scenario: Успешное создание пространства

- **WHEN** пользователь отправляет `/newspace` и вводит название "Семья"
- **THEN** бот создаёт пространство с названием "Семья", генерирует invite-код, добавляет пользователя как admin, отвечает сообщением с названием и invite-ссылкой вида `https://t.me/<bot_username>?start=join_<invite_code>`

#### Scenario: Название пространства слишком длинное

- **WHEN** пользователь вводит название длиннее 255 символов
- **THEN** бот отвечает сообщением об ошибке и просит ввести название покороче, FSM остаётся в состоянии ожидания названия

#### Scenario: Пустое название пространства

- **WHEN** пользователь отправляет пустое сообщение или пробелы в ответ на запрос названия
- **THEN** бот отвечает сообщением об ошибке и просит ввести название

#### Scenario: FSM-состояние при /newspace

- **WHEN** пользователь отправляет `/newspace`
- **THEN** бот устанавливает FSM-состояние `CreateSpace.waiting_for_name` и отправляет сообщение "Введи название пространства:"

### Requirement: Список пространств через /spaces

Команда `/spaces` ДОЛЖНА показывать список пространств пользователя с названием, ролью и количеством участников.

#### Scenario: Пользователь состоит в нескольких пространствах

- **WHEN** пользователь, состоящий в пространствах "Семья" (admin, 3 участника) и "Работа" (member, 5 участников), отправляет `/spaces`
- **THEN** бот отвечает списком: для каждого пространства — название, роль пользователя, количество участников

#### Scenario: Пользователь не состоит ни в одном пространстве

- **WHEN** пользователь, не состоящий ни в одном пространстве, отправляет `/spaces`
- **THEN** бот отвечает сообщением "Ты пока не состоишь ни в одном пространстве. Создай новое через /newspace"

### Requirement: Информация о пространстве через /space_info

Команда `/space_info` ДОЛЖНА показывать детальную информацию: название, список участников (имя, роль), invite-ссылку. Если пользователь состоит в нескольких пространствах — ДОЛЖНА предложить выбрать через inline-клавиатуру.

#### Scenario: Пользователь в одном пространстве

- **WHEN** пользователь, состоящий в одном пространстве "Семья", отправляет `/space_info`
- **THEN** бот показывает информацию о "Семья": список участников с ролями и invite-ссылку

#### Scenario: Пользователь в нескольких пространствах

- **WHEN** пользователь, состоящий в нескольких пространствах, отправляет `/space_info`
- **THEN** бот показывает inline-клавиатуру с выбором пространства

#### Scenario: Выбор пространства для info через inline-кнопку

- **WHEN** пользователь нажимает кнопку пространства в клавиатуре выбора для `/space_info`
- **THEN** бот показывает информацию о выбранном пространстве: список участников с ролями и invite-ссылку

### Requirement: Удаление участника через /kick

Команда `/kick @username` ДОЛЖНА удалять участника из пространства. ДОЛЖНА быть доступна только администратору пространства. Если пространств несколько — ДОЛЖНА предложить выбрать.

#### Scenario: Администратор удаляет участника

- **WHEN** администратор пространства отправляет `/kick @username` и указанный пользователь является участником этого пространства
- **THEN** запись `user_spaces` для указанного пользователя удаляется, бот подтверждает удаление

#### Scenario: Не-админ пытается кикнуть

- **WHEN** пользователь с ролью member отправляет `/kick @username`
- **THEN** бот отвечает сообщением об ошибке: "Только администратор может удалять участников"

#### Scenario: Кик несуществующего участника

- **WHEN** администратор отправляет `/kick @username`, но пользователь с таким username не найден в пространстве
- **THEN** бот отвечает сообщением: "Пользователь не найден в этом пространстве"

#### Scenario: Администратор пытается кикнуть себя

- **WHEN** администратор отправляет `/kick` со своим собственным username
- **THEN** бот отвечает сообщением об ошибке: "Нельзя удалить себя из пространства"

### Requirement: Удаление пространства через /delete_space

Команда `/delete_space` ДОЛЖНА удалять пространство с каскадным удалением всех событий и напоминаний. ДОЛЖНА быть доступна только администратору. ДОЛЖНА требовать подтверждение через inline-кнопки.

#### Scenario: Администратор удаляет пространство с подтверждением

- **WHEN** администратор отправляет `/delete_space`, выбирает пространство (если несколько) и нажимает кнопку подтверждения
- **THEN** пространство удаляется из БД вместе со всеми `user_spaces`, `events`, `scheduled_reminders`, бот подтверждает удаление

#### Scenario: Администратор отменяет удаление

- **WHEN** администратор нажимает кнопку отмены в подтверждении удаления
- **THEN** пространство не удаляется, бот сообщает об отмене

#### Scenario: Не-админ пытается удалить пространство

- **WHEN** пользователь с ролью member отправляет `/delete_space`
- **THEN** бот отвечает сообщением: "Только администратор может удалить пространство"

### Requirement: Inline-клавиатура выбора пространства

Система ДОЛЖНА предоставлять переиспользуемую inline-клавиатуру для выбора пространства. Callback data ДОЛЖЕН иметь формат `space_select:{space_id}:{action}`.

#### Scenario: Генерация клавиатуры выбора

- **WHEN** пользователю нужно выбрать пространство для действия (info, kick, delete, и т.д.)
- **THEN** генерируется inline-клавиатура с кнопками — по одной на каждое пространство пользователя, callback data содержит ID пространства и действие

### Requirement: Сервисный слой space_service.py

Вся бизнес-логика пространств ДОЛЖНА быть в `services/space_service.py` как async-функции, принимающие `session: AsyncSession`. Handlers НЕ ДОЛЖНЫ содержать прямых запросов к БД.

#### Scenario: Handler вызывает сервис для создания пространства

- **WHEN** handler обрабатывает создание пространства
- **THEN** handler вызывает `space_service.create_space(session, user_id, name)`, который создаёт записи в `spaces` и `user_spaces` и возвращает созданное пространство с invite-ссылкой

### Requirement: Фильтр not_command

Модуль `filters/not_command.py` ДОЛЖЕН предоставлять фильтр, определяющий, что текстовое сообщение не является командой бота (не начинается с `/`).

#### Scenario: Обычное текстовое сообщение проходит фильтр

- **WHEN** пользователь отправляет сообщение "Привет"
- **THEN** фильтр `not_command` возвращает True

#### Scenario: Команда не проходит фильтр

- **WHEN** пользователь отправляет сообщение `/start`
- **THEN** фильтр `not_command` возвращает False
