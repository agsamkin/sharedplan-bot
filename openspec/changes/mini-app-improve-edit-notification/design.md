## Context

При редактировании/удалении события через Mini App уведомления формируются инлайн-строками прямо в `mini_app/backend/routes/events.py`:

- Редактирование (строка 295): `f"📝 Событие изменено: {updated_event.title}"` — нет информации о том, что изменилось, какие были старые/новые значения, кто изменил, в каком пространстве.
- Удаление (строка 354): `f"🗑 Событие удалено: {event.title}"` — нет даты/времени удалённого события, имени удалившего, названия пространства.

При этом в `app/bot/formatting.py` уже есть готовые функции:
- `format_event_edited_notification(space_name, title, field_label, old_value, new_value, editor_name)` — детальное уведомление с указанием изменённого поля и значений «до → после».
- `format_event_deleted_notification(space_name, title, editor_name)` — уведомление с названием пространства и именем удалившего.

Обе функции не импортируются и не вызываются нигде в кодовой базе.

## Goals / Non-Goals

**Goals:**
- Уведомления при редактировании содержат: название пространства, название события, что изменилось (старое → новое значение), имя редактора
- Уведомления при удалении содержат: название пространства, название события, дату/время удалённого события, имя удалившего
- Единообразный формат уведомлений из Mini App и бота
- Переиспользование существующих функций форматирования

**Non-Goals:**
- Изменение UX самого Mini App (формы редактирования, кнопки)
- Изменение логики напоминаний
- Добавление push-уведомлений или каких-либо новых каналов доставки
- История изменений / аудит-лог

## Decisions

### 1. Переиспользование `format_event_edited_notification` для каждого изменённого поля

**Решение**: при редактировании события через Mini App — определить, какие поля изменились (title, event_date, event_time), и для каждого изменённого поля вызвать `format_event_edited_notification` с соответствующим `field_label`, `old_value`, `new_value`. Все изменённые поля объединяются в одно сообщение.

**Проблема текущей функции**: `format_event_edited_notification` принимает одно поле (`field_label`, `old_value`, `new_value`). При редактировании через Mini App пользователь может изменить несколько полей одновременно (например, дату и время).

**Решение**: доработать `format_event_edited_notification` — принимать список изменений вместо одного поля. Формат:

```
✏️ Событие изменено в «Семья»!

📝 Ужин с друзьями
🔄 Дата: 28 марта, суббота → 30 марта, понедельник
🔄 Время: 19:00 → 20:30
👤 Изменил: Алексей
```

**Альтернатива**: отправлять отдельное уведомление на каждое изменённое поле. Отклонена: создаёт спам при одновременном изменении нескольких полей.

### 2. Сохранение старых значений перед обновлением

**Решение**: в `update_event` handler Mini App backend — сохранить старые значения полей события ДО вызова `event_service.update_event`, чтобы иметь возможность сравнить и отформатировать diff.

### 3. Формат значений в уведомлении

**Решение**: использовать существующие функции форматирования дат из `formatting.py`:
- Дата: `format_date_short_with_weekday` (например, «28 марта, суббота») — информативно и компактно
- Время: `HH:MM` (например, «19:00»)
- Название: как есть, в кавычках при необходимости
- Для события без времени: «весь день»

### 4. Обогащение уведомления об удалении

**Решение**: заменить инлайн-строку в `delete_event` handler на вызов `format_event_deleted_notification`. Дополнительно добавить в функцию дату/время события, чтобы участники понимали, какое именно событие было удалено.

Новый формат:

```
🗑 Событие удалено в «Семья»!

📝 Ужин с друзьями
📅 28 марта, суббота, 19:00
👤 Удалил: Алексей
```

### 5. Импорт formatting-модуля в Mini App backend

**Решение**: `mini_app/backend/routes/events.py` уже импортирует из `app.services` и `app.db.models` — добавление импорта из `app.bot.formatting` не нарушает архитектуру, т.к. formatting-модуль не зависит от Telegram-специфичных библиотек (чистые функции форматирования строк).

## Risks / Trade-offs

- **Изменение сигнатуры `format_event_edited_notification`** → функция пока не используется нигде, поэтому изменение сигнатуры безопасно. Обратная совместимость не требуется.
- **Изменение сигнатуры `format_event_deleted_notification`** → аналогично, функция не используется. Добавление опциональных параметров (event_date, event_time) сохранит обратную совместимость.
- **Длинное уведомление при изменении всех трёх полей** → Маловероятный сценарий; даже при изменении всех полей уведомление остаётся компактным (5-6 строк).
