## Context

Участник пространства не видит события, созданные владельцем. Анализ кода показал, что SQL-запросы корректны — `get_upcoming_events` фильтрует по `space_id`, а не по `created_by`. API-эндпоинт `list_events` проверяет членство и возвращает все события пространства. Фронтенд корректно вызывает API и отображает результат.

Возможные причины бага:
1. **Жёсткий лимит `limit=10`** — если в пространстве более 10 предстоящих событий, часть не отображается, причём какие именно — зависит от сортировки, а не от создателя.
2. **Фильтрация прошедших событий** — `get_upcoming_events` исключает прошедшие. Если событие на сегодня и время уже прошло, оно исчезает.
3. **Ошибка авторизации** — если `initData` участника невалиден или просрочен (>1 час), API вернёт 401, а фронтенд покажет ошибку. Но пользователь может интерпретировать это как «события не видны».
4. **Проблема с членством** — если `UserSpace`-запись участника отсутствует или повреждена, API вернёт 403.

Поскольку точная причина требует данных продакшена, исправление должно быть направлено на улучшение наблюдаемости и устранение наиболее вероятных ограничений.

## Goals / Non-Goals

**Goals:**
- Добавить диагностическое логирование в эндпоинт списка событий для отладки проблем видимости
- Убрать жёсткий лимит 10 событий, заменив на параметр запроса с пагинацией
- Добавить `total_count` в ответ API для диагностики

**Non-Goals:**
- Менять логику фильтрации событий (она корректна)
- Менять авторизацию или модель данных
- Добавлять новые эндпоинты

## Decisions

### 1. Параметризация лимита событий

**Решение:** Заменить хардкод `limit=10` на query-параметр `limit` (default=50, max=100).

**Альтернатива:** Убрать лимит полностью — отвергнуто, т.к. пространство может накопить тысячи событий.

**Обоснование:** 10 — слишком мало для активных пространств. 50 — разумный default. Max=100 защищает от злоупотреблений.

### 2. Добавление `total_count` в ответ

**Решение:** Добавить поле `total_count` в JSON-ответ эндпоинта `GET /api/spaces/:space_id/events` — общее количество предстоящих событий в пространстве (без лимита). Ответ станет объектом `{ events: [...], total_count: N }` вместо массива.

**Альтернатива:** Отдельный эндпоинт для подсчёта — отвергнуто, лишний запрос.

**Обоснование:** Позволяет фронтенду показать «События · 5 из 50» и обнаружить расхождения.

### 3. Структурное логирование запросов событий

**Решение:** Добавить `logger.info(...)` в `list_events` с данными: `user_id`, `space_id`, количество возвращённых событий, `total_count`.

**Обоснование:** Позволит быстро диагностировать проблемы видимости в логах без дополнительного инструментария.

### 4. Обновление фронтенда

**Решение:** Обновить `SpaceDetailPage` для работы с новым форматом ответа (объект вместо массива). Отображать `total_count` в заголовке секции событий.

## Risks / Trade-offs

- **[Ломающее изменение API]** → Ответ `GET /api/spaces/:space_id/events` меняется с массива на объект. Это требует одновременного обновления фронтенда и бэкенда. Mitigation: деплой через единый Docker-образ.
- **[Производительность COUNT]** → Добавляется дополнительный COUNT-запрос. Mitigation: запрос по индексу `idx_events_space_date`, пренебрежимо мало на типичных объёмах.
