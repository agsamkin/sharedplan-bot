## Context

Удаление события в Mini App реализовано полным стеком: кнопка «Удалить событие» в `EventDetailPage.tsx` → подтверждение через `ConfirmInline` → вызов `deleteEvent(id)` в `api/events.ts` → `del<void>('/api/events/' + id)` в `api/client.ts` → `fetch(url, { method: 'DELETE', headers: { 'Content-Type': 'application/json', 'Authorization': ... } })`.

Бэкенд (`mini_app/backend/routes/events.py`) корректно обрабатывает DELETE: проверяет владение, уведомляет участников, удаляет событие через `event_service.delete_event`, возвращает 204. Middleware (`db_session_middleware`) коммитит транзакцию после ответа хэндлера.

Проблема на стороне фронтенда состоит из двух частей:
1. **Лишний `Content-Type: application/json`** — клиент всегда устанавливает этот заголовок, даже для DELETE-запросов без тела. В Telegram WebView (на базе Android WebView / WKWebView) это может приводить к тому, что браузер ожидает тело запроса или корректирует запрос непредсказуемо.
2. **Ненадёжная навигация** — `navigate(-1)` возвращает на предыдущую запись в history stack. В Telegram WebView компонент может не перемонтироваться (bfcache), и `useEffect` с `fetchData` не срабатывает повторно. Пользователь видит устаревший список событий.

## Goals / Non-Goals

**Goals:**
- Исправить отправку DELETE-запроса: убрать `Content-Type` при отсутствии тела
- Обеспечить надёжное обновление списка событий после удаления
- Добавить визуальную обратную связь при удалении (состояние загрузки)
- Показывать toast-уведомление при ошибке удаления

**Non-Goals:**
- Изменение бэкенд-логики удаления (она корректна)
- Рефакторинг других API-вызовов (PUT, POST) — они работают корректно с телом запроса
- Глобальное кэширование/инвалидация данных (overengineering для текущего масштаба)
- Добавление retry-логики при неудачном удалении

## Decisions

### 1. Условный Content-Type в API-клиенте

**Решение**: устанавливать заголовок `Content-Type: application/json` только когда `body !== undefined`. Для DELETE-запросов без тела заголовок не отправляется.

**Альтернатива**: передавать пустой объект `{}` как тело DELETE-запроса. Отклонена: семантически некорректно, DELETE не должен иметь тела по RFC 7231.

**Файл**: `mini_app/frontend/src/api/client.ts`, функция `request()`.

### 2. Навигация через replace вместо navigate(-1)

**Решение**: после успешного удаления события использовать `navigate('/spaces/' + spaceId, { replace: true })` вместо `navigate(-1)`. Для этого страница EventDetailPage загружает `space_id` из ответа API `getEvent` (уже возвращается бэкендом в поле `space_id`).

**Альтернатива 1**: добавить `key` к маршруту SpaceDetailPage для принудительного ремонтирования. Отклонена: хак, не решает корневую проблему.

**Альтернатива 2**: использовать глобальный state-менеджер для инвалидации кэша. Отклонена: overengineering для одного бага.

**Файлы**: `mini_app/frontend/src/pages/EventDetailPage.tsx`.

### 3. Состояние загрузки при удалении

**Решение**: добавить state `deleting` (boolean). Во время удаления кнопка «Удалить» в ConfirmInline заменяется на «Удаление...» и становится disabled. Это предотвращает повторные нажатия и даёт визуальную обратную связь.

**Файл**: `mini_app/frontend/src/pages/EventDetailPage.tsx`.

### 4. Toast при ошибке удаления

**Решение**: в блоке `catch` вызывать `showToast(message)` помимо `setError(message)`, чтобы пользователь гарантированно увидел уведомление (toast заметнее, чем текст ошибки внизу страницы).

**Файл**: `mini_app/frontend/src/pages/EventDetailPage.tsx`.

## Risks / Trade-offs

- **Удаление `Content-Type` для запросов без тела может теоретически затронуть другие DELETE-вызовы** → `deleteSpace` и `removeMember` в `api/spaces.ts` тоже используют `del()` без тела — они тоже получат исправление, что корректно.
- **Навигация по абсолютному пути требует знания `space_id`** → `space_id` уже возвращается бэкендом в ответе `GET /api/events/{event_id}`, нужно только сохранить его в state компонента.
