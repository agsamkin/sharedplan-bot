## Context

При создании события пользователь не получает информации о существующих событиях в пространстве на близкое время. Уже есть аналогичный паттерн — предупреждение о прошедшей дате (`_is_event_in_past` → `waiting_for_past_confirm`). Новое предупреждение о конфликте времени встраивается в тот же флоу.

Текущий флоу подтверждения:
1. Парсинг → выбор пространства → проверка прошедшей даты → карточка подтверждения
2. Mini App: PUT `/api/events/{id}` — редактирование без проверки конфликтов

## Goals / Non-Goals

**Goals:**
- Предупреждать пользователя, если в пространстве уже есть событие в окне +-2 часа от нового
- Показывать список конфликтующих событий в предупреждении
- Не блокировать создание — пользователь может подтвердить несмотря на конфликт
- Работать для обоих каналов: бот (текст/голос) и Mini App (при создании и редактировании)

**Non-Goals:**
- Проверка конфликтов между разными пространствами
- Автоматическое предложение альтернативного времени
- Настраиваемый размер окна конфликта (захардкожены 2 часа)
- Проверка конфликтов для событий без времени (`event_time IS NULL`)

## Decisions

### 1. Сервисная функция `find_conflicting_events` в `event_service.py`

Новая функция выполняет запрос к таблице `events` в рамках одного пространства. Фильтрация: та же дата, время в окне +-2 часа. Возвращает список конфликтующих событий.

**Почему в event_service:** Следуем layer rules — бизнес-логика в сервисах, не в хендлерах. Функция переиспользуется и в боте, и в Mini App.

**Обработка граничных случаев:**
- События без времени (`event_time IS NULL`) **не считаются конфликтующими** — они «на весь день», конкретного пересечения нет
- Новое событие без времени **не проверяется** на конфликты — аналогичная логика
- Пересечение через полночь (напр. 23:00 + 2ч = 01:00 следующего дня): проверяем только в рамках одной даты. Для MVP это приемлемое упрощение

### 2. Интеграция в флоу бота — после проверки прошедшей даты, перед карточкой подтверждения

Конфликт проверяется в `_show_confirmation_or_past_warning` (переименовываем в `_show_confirmation_with_warnings`). Порядок проверок:
1. Прошедшая дата → предупреждение (существующее)
2. Конфликт времени → предупреждение (новое)
3. Нет проблем → карточка подтверждения

**Почему не новый FSM-стейт:** Предупреждение о конфликте — информационное дополнение к карточке подтверждения, а не блокирующий шаг. Текст предупреждения добавляется в начало карточки подтверждения. Это проще и не усложняет FSM.

**Альтернатива (отвергнута):** Отдельный стейт `waiting_for_conflict_confirm` по аналогии с past-date. Отвергнута — нет необходимости в отдельном шаге подтверждения, достаточно показать информацию.

### 3. Формат предупреждения в карточке подтверждения

Текст предупреждения добавляется перед основной карточкой:
```
⚠️ На близкое время уже есть события:
• Ужин с друзьями (19:00)
• Звонок маме (20:30)

📌 Событие:
📝 Встреча
📅 28 марта 2026, суббота
⏰ 18:00

Опубликовать?
```

Кнопки остаются стандартные: `[✅ Да]` `[✏️ Изменить]` `[❌ Отмена]`.

### 4. Mini App — поле `conflicts` в ответе при создании/редактировании

Mini App не создаёт события напрямую (создание идёт через бота). Но при **редактировании** (PUT `/api/events/{id}`) с изменением даты/времени — добавляем поле `conflicts` в ответ с массивом конфликтующих событий. Фронтенд может показать предупреждение.

## Risks / Trade-offs

- **[Упрощение полночи]** Не проверяем конфликты через границу суток (23:00 vs 01:00 следующего дня) → Приемлемо для MVP, можно добавить позже
- **[Дополнительный запрос к БД]** Каждое создание события делает доп. запрос → Запрос лёгкий (индекс `idx_events_space_date` по `space_id + event_date`), влияние минимально
- **[Информационное предупреждение vs блокирующее]** Пользователь может не заметить предупреждение → Но блокирующий шаг усложняет UX для легитимных случаев (два события рядом — нормально)
