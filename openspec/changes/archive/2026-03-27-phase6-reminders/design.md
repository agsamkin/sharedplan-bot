## Context

Фазы 1–5 реализованы: бот работает с пространствами, событиями (текст + голос + AI-парсинг), уведомлениями участников. Таблица `scheduled_reminders` и поле `users.reminder_settings` уже существуют в БД (созданы в Фазе 1), но не используются — при подтверждении события напоминания не создаются, планировщик не запущен, команда `/reminders` не реализована.

Текущий `event_confirm.py` создаёт событие через `event_service.create_event()` и отправляет уведомления участникам. APScheduler и `scheduler/jobs.py` ещё не существуют. Конфигурация `REMINDER_CHECK_INTERVAL_SECONDS` уже определена в `config.py`.

## Goals / Non-Goals

**Goals:**
- Автоматическое создание записей `scheduled_reminders` при публикации события
- Фоновая задача для отправки наступивших напоминаний каждые N секунд
- Команда `/reminders` с inline toggle-интерфейсом для настройки интервалов
- Корректная обработка событий без времени (только напоминание `1d` на 09:00 накануне)
- Изоляция ошибок: сбой отправки одного напоминания не блокирует остальные

**Non-Goals:**
- Per-event настройки напоминаний (только глобальные на пользователя)
- Ретроактивный пересчёт напоминаний при изменении настроек
- Retry для неотправленных напоминаний (sent=TRUE при ошибке)
- Поддержка нескольких часовых поясов
- Удаление/редактирование уже созданных напоминаний при удалении события (CASCADE на уровне БД уже обеспечивает это)

## Decisions

### 1. Создание напоминаний в event_service, а не в callback-хендлере

Логика создания `scheduled_reminders` добавляется в `services/reminder_service.py` как функция `create_reminders_for_event()`, вызываемая из `event_confirm.py` после `event_service.create_event()`. Бизнес-логика расчёта `remind_at` остаётся в сервисном слое, Telegram-специфичный код — в хендлере.

**Альтернатива**: Встроить создание напоминаний прямо в `event_service.create_event()`. Отклонено — это смешивает ответственности (event CRUD vs reminder scheduling) и усложняет тестирование.

### 2. Расчёт remind_at в часовом поясе Europe/Moscow

Все вычисления `remind_at` используют `Europe/Moscow` (из `settings.TIMEZONE`). Для события с датой `2026-04-05` и временем `19:00`:
- `1d`: `2026-04-04T19:00:00+03:00`
- `2h`: `2026-04-05T17:00:00+03:00`
- `1h`: `2026-04-05T18:00:00+03:00`
- `30m`: `2026-04-05T18:30:00+03:00`
- `15m`: `2026-04-05T18:45:00+03:00`
- `0m`: `2026-04-05T19:00:00+03:00`

Для событий без времени — только `1d`, `remind_at` = `09:00 накануне`.

Фильтрация: если `remind_at <= now()`, напоминание не создаётся.

### 3. APScheduler как AsyncIOScheduler в том же event loop

Планировщик запускается в `main.py` перед `dp.start_polling()`. Одна интервальная задача `process_due_reminders` с интервалом `REMINDER_CHECK_INTERVAL_SECONDS`.

Задача создаёт собственную DB-сессию через `async_session()` (не через middleware), потому что выполняется вне контекста aiogram-обработчика.

**Альтернатива**: Передавать `Bot` instance через аргументы задачи APScheduler. Принято — `bot` передаётся при регистрации задачи.

### 4. Отправка напоминаний пакетами по 50

Запрос: `WHERE sent = FALSE AND remind_at <= NOW() ORDER BY remind_at LIMIT 50`. Каждое напоминание помечается `sent=TRUE` индивидуально после попытки отправки (успешной или нет). Если наступивших > 50 — следующий тик подберёт остаток.

### 5. Toggle-интерфейс /reminders через edit_message

Команда `/reminders` отправляет сообщение с текущими настройками и inline-клавиатурой. Каждое нажатие toggle-кнопки:
1. Читает текущие `reminder_settings` пользователя из БД
2. Инвертирует выбранный ключ
3. Записывает обновлённые настройки
4. Вызывает `edit_message_reply_markup` для обновления клавиатуры

Callback data формат: `reminder_toggle:{key}` (например, `reminder_toggle:1d`).

### 6. Формат сообщения напоминания

```
🔔 Напоминание!

📝 {title}
⏰ {relative_label} ({time})
📍 Пространство: {space_name}
```

Где `relative_label` зависит от `reminder_type`:
- `1d` → «Завтра»
- `2h` → «Через 2 часа»
- `1h` → «Через 1 час»
- `30m` → «Через 30 минут»
- `15m` → «Через 15 минут»
- `0m` → «Сейчас»

Для событий без времени (только `1d`): строка ⏰ заменяется на `📅 Завтра`.

## Risks / Trade-offs

**[Планировщик работает в том же процессе]** → Если event loop заблокирован, напоминания задерживаются. Mitigation: все операции async, CPU-bound работы нет.

**[sent=TRUE при ошибке отправки]** → Пользователь может не получить напоминание, если бот был заблокирован. Mitigation: осознанное решение — альтернатива (retry) создаёт растущую очередь неотправляемых напоминаний. Ошибки логируются.

**[Напоминания не пересчитываются при смене настроек]** → Если пользователь изменит настройки после публикации события, существующие напоминания не обновятся. Mitigation: документировано в UX — «Изменения применяются к будущим событиям».

**[Race condition при массовом создании событий]** → Маловероятно при масштабе личного бота (< 50 пользователей). Mitigation: каждое создание события — отдельная транзакция.
