## Context

Фаза 1 реализована: бот стартует в Docker, БД с 5 таблицами создана через Alembic, `/start` делает upsert пользователя, `/help` возвращает список команд. Middleware внедряет DB-сессию в handlers.

Таблицы `spaces` и `user_spaces` уже существуют в БД — нужно только написать логику работы с ними. Новых миграций не требуется.

Текущая структура: `app/bot/handlers/` содержит `start.py` и `help.py`, роутеры подключаются в `main.py`. Сервисного слоя пока нет.

## Goals / Non-Goals

**Goals:**

- Полный жизненный цикл пространств: создание → invite-ссылка → присоединение → просмотр → администрирование → удаление
- Чёткое разделение слоёв: handlers (Telegram-логика) → services (бизнес-логика) → db/models (данные)
- Deep link работает из Telegram — пользователь нажимает ссылку и попадает в бота с автоматическим присоединением
- Выбор пространства через inline-клавиатуру (для команд, работающих с конкретным пространством, когда пользователь в нескольких)

**Non-Goals:**

- События и их создание (Фаза 3)
- AI-парсинг (Фаза 4)
- Напоминания (Фаза 6)
- Редактирование названия пространства
- Передача прав администратора
- Пагинация списка пространств

## Decisions

### 1. Сервисный слой `space_service.py` — stateless-функции

Вся бизнес-логика в `services/space_service.py` как набор async-функций, принимающих `session: AsyncSession` первым аргументом. Не класс — нет состояния, нечего инстанцировать.

**Альтернатива**: класс `SpaceService` с DI сессии через конструктор. Отклонено — для текущего масштаба излишне, функции проще и прямолинейнее.

### 2. FSM для создания пространства — одно состояние

`CreateSpace.waiting_for_name` — единственное состояние FSM. `/newspace` ставит состояние и спрашивает название, следующее текстовое сообщение — название пространства. Используем встроенный FSM aiogram с MemoryStorage (состояние не переживает рестарт — допустимо).

**Альтернатива**: без FSM, название в той же команде (`/newspace Семья`). Отклонено — противоречит UX из концепции (бот спрашивает название).

### 3. Deep link в `/start` — ветвление по payload

Расширяем обработчик `/start`: если `CommandStart` содержит `deep_link` с префиксом `join_`, извлекаем invite_code и запускаем флоу присоединения. Иначе — стандартное приветствие.

aiogram 3.x поддерживает `CommandStart(deep_link=True)` и `command.args` для получения payload.

Реализация: два отдельных обработчика в `start.py`:
- `cmd_start_join` с фильтром `CommandStart(deep_link=True, deep_link_encoded=True)` и кастомным фильтром на `join_` префикс — обрабатывает присоединение
- `cmd_start` — fallback, обычное приветствие

**Альтернатива**: один обработчик с `if/else` внутри. Отклонено — два обработчика чище, aiogram сам маршрутизирует по фильтрам.

### 4. Выбор пространства — переиспользуемая inline-клавиатура

`keyboards/space_select.py` генерирует inline-клавиатуру со списком пространств пользователя. Callback data: `space_select:{space_id}:{action}`, где `action` — строка, определяющая контекст (например `info` для `/space_info`). Это позволит переиспользовать клавиатуру в Фазе 3 (выбор пространства для события).

`callbacks/space_select.py` парсит callback, вызывает соответствующий handler в зависимости от `action`.

### 5. Invite-код — `secrets.token_urlsafe(6)`

Генерирует 8-символьную URL-safe строку. Уникальность обеспечивается UNIQUE-ограничением в БД. При коллизии (крайне маловероятно) — retry с новым кодом.

Invite-ссылка формируется как `https://t.me/{bot_username}?start=join_{invite_code}`. `bot_username` получаем через `bot.me.username` (доступен после `bot.get_me()`, aiogram кэширует).

### 6. `/kick` — по username в аргументе команды

`/kick @username` в чате с ботом. Handler парсит username из `message.text`, ищет пользователя в пространстве. Требует выбор пространства если их несколько.

**Альтернатива**: inline-кнопки со списком участников для удаления. Возможно удобнее, но усложняет — оставим текстовую команду для MVP.

### 7. `/delete_space` — inline-подтверждение

Показываем сообщение «Удалить пространство {name}? Все события и напоминания будут удалены.» с кнопками `[Да, удалить]` `[Отмена]`. Callback data: `delete_space_confirm:{space_id}` / `delete_space_cancel:{space_id}`.

### 8. Уведомление участников при join — fire-and-forget

При присоединении нового участника — отправляем сообщение каждому текущему участнику пространства. Каждая отправка обёрнута в try/except. Ошибки логируются, не прерывают процесс.

### 9. Получение bot_username

Для формирования invite-ссылки нужен username бота. Передаём объект `Bot` в handlers через middleware или `data["bot"]` (aiogram автоматически внедряет `bot` в handler data). Используем `(await bot.get_me()).username` — aiogram кэширует результат.

## Risks / Trade-offs

- **FSM-состояние теряется при рестарте** → Допустимо: пользователь просто начнёт `/newspace` заново. MemoryStorage — осознанный выбор из архитектуры проекта.
- **`/kick` требует знать username** → Пользователь без username не может быть удалён через `/kick @username`. Митигация: `/space_info` показывает список участников, в будущем можно добавить inline-кнопки.
- **Нет лимита на количество пространств у пользователя** → На текущем масштабе (~20 пространств) не проблема. Можно добавить soft limit позже.
- **Гонка при join: два пользователя одновременно** → Составной PK `(user_id, space_id)` гарантирует уникальность на уровне БД. При дублировании — IntegrityError, обрабатываем как «уже участник».
