## Context

Фазы 1–2 реализованы: бот стартует, управляет пространствами (создание, invite, kick, delete), пользователи присоединяются по deep link. Модели `Event` и `ScheduledReminder` уже существуют в `db/models.py`, но никакой код их не использует. Фильтр `NotCommandFilter` готов в `filters/not_command.py`. Keyboard `space_select.py` поддерживает выбор пространства с параметром action.

Фаза 3 добавляет создание событий с заглушкой парсинга (структурированный формат вместо AI), подтверждение через inline-кнопки и уведомление участников.

## Goals / Non-Goals

**Goals:**
- Полный цикл создания события: ввод текста → парсинг → подтверждение → сохранение → уведомления
- Команда `/events` для просмотра ближайших событий пространства
- Заглушка парсинга, которую Фаза 4 заменит на LLM без изменения интерфейса
- Изоляция ошибок при уведомлениях (сбой одной отправки не прерывает остальные)

**Non-Goals:**
- AI-парсинг естественного языка (Фаза 4)
- Голосовые сообщения (Фаза 5)
- Напоминания и `scheduled_reminders` (Фаза 6)
- Удаление или редактирование существующих событий
- Пагинация в `/events`

## Decisions

### 1. Формат заглушки парсинга

**Решение:** Пользователь вводит событие в формате `название, дата, время` (через запятую). Дата — `ДД.ММ.ГГГГ`. Время — `ЧЧ:ММ` (опционально). Примеры: `Ужин с родителями, 27.03.2026, 19:00` или `День рождения Ани, 05.04.2026`.

**Почему:** Простой формат, не требующий AI. Парсинг через `split(",")` + `datetime.strptime`. Фаза 4 заменит этот парсер на LLM-вызов, сохраняя тот же интерфейс: `parse_event(text) → {title, date, time}`.

**Альтернатива:** Regex-парсинг свободного текста — отвергнут, т.к. ненадёжен и дублирует работу Фазы 4.

### 2. Архитектура парсера — функция с заменяемым интерфейсом

**Решение:** Парсер реализуется как async-функция `parse_event_text(text: str) → ParsedEvent` в `services/event_service.py`. `ParsedEvent` — dataclass/NamedTuple с полями `title: str`, `date: date`, `time: Optional[time]`.

**Почему:** Фаза 4 заменит тело функции на LLM-вызов. Хендлер не изменится — он вызывает `parse_event_text()` и получает `ParsedEvent`.

### 3. Хранение черновика события

**Решение:** Черновик (parsed event) хранится в FSM-данных aiogram (`state.update_data()`), не в БД. В БД событие попадает только после подтверждения.

**Почему:** Черновик — временное состояние. Нет смысла создавать запись в БД с флагом `draft`. FSM-данные aiogram живут в памяти — при перезагрузке бота черновик теряется, что приемлемо.

**Альтернатива:** Хранение в callback_data — отвергнуто из-за лимита 64 байта на callback_data в Telegram.

### 4. Выбор пространства при создании события

**Решение:** Переиспользуем существующую клавиатуру `space_select_keyboard` с action `"event"`. Callback `space_select:{space_id}:event` сохраняет выбранное пространство в FSM-данные и показывает карточку подтверждения.

**Почему:** Клавиатура уже умеет показывать список пространств. Нужно только добавить обработку action `"event"` в callback-хендлере.

### 5. Callback data для подтверждения события

**Решение:** Три кнопки с callback data: `event_confirm`, `event_edit`, `event_cancel`. Без UUID в callback data — контекст берётся из FSM-состояния текущего пользователя.

**Почему:** UUID черновика не существует (событие ещё не в БД). FSM-состояние однозначно связано с пользователем. Один пользователь не может создавать два события параллельно.

### 6. Уведомление участников

**Решение:** После сохранения события — запрос участников пространства через `space_service.get_space_members()`, отправка каждому (кроме создателя) личного сообщения через `bot.send_message()`. Каждая отправка обёрнута в `try/except`.

**Почему:** Требование изоляции ошибок из архитектурного документа. Если пользователь заблокировал бота — логируем и продолжаем.

### 7. Относительные даты в /events

**Решение:** Даты «сегодня» и «завтра» отображаются как относительные. Остальные — числовым форматом (`5 апреля`). Логика в хендлере, не в сервисе.

**Почему:** Форматирование — ответственность Telegram-слоя (handlers), не бизнес-логики (services).

## Risks / Trade-offs

- **FSM-состояние теряется при перезапуске бота** → Допустимо для личного бота. Пользователь просто повторяет ввод. Миграция: нет.
- **Заглушка парсинга требует точный формат** → Пользователь видит инструкцию в сообщении ошибки. Фаза 4 полностью устранит проблему.
- **Нет проверки на дублирование событий** → Осознанное упрощение. Одно и то же событие можно создать дважды. Для личного бота это приемлемо.
- **Один callback-handler для space_select обрабатывает все actions** → Растущий switch/case. При необходимости рефакторить в отдельные handler-ы. Сейчас actions: info, kick, delete + event — терпимо.
