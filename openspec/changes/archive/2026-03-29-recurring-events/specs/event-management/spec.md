## MODIFIED Requirements

### Requirement: Event deletion
Владелец события ДОЛЖЕН иметь возможность удалить событие через диалог подтверждения. DELETE-запрос НЕ ДОЛЖЕН отправлять `Content-Type` при пустом теле. Показать loading-состояние, при успехе перейти на `/spaces/{space_id}` с `replace: true`. Toast при успехе/ошибке. Уведомить участников через бот. 404 → toast "Событие не найдено". Если событие повторяющееся (`recurrence_rule != null`), текст диалога ДОЛЖЕН предупреждать: "Удалить «{title}» и все будущие повторения? Это действие нельзя отменить."

#### Scenario: Удаление повторяющегося события
- **WHEN** владелец удаляет событие с `recurrence_rule != null`
- **THEN** диалог предупреждает о удалении серии, при подтверждении удаляются родительское событие и все дочерние вхождения каскадно

#### Scenario: Удаление одноразового события
- **WHEN** владелец удаляет событие с `recurrence_rule = null`
- **THEN** стандартный диалог без упоминания серии

### Requirement: Title editing
Редактирование заголовка через форму, валидация ≤500 символов, `PUT /api/events/:id`, уведомление участников. Если событие повторяющееся — обновить заголовок у всех будущих дочерних вхождений.

#### Scenario: Редактирование заголовка повторяющегося события
- **WHEN** владелец меняет title повторяющегося события
- **THEN** title обновляется у родительского события и всех дочерних вхождений с `event_date >= today`

### Requirement: Recurrence editing
Владелец ДОЛЖЕН иметь возможность изменить `recurrence_rule` события через RepeatPicker. При изменении правила повторения все будущие дочерние вхождения удаляются и генерируются заново. При изменении с повторяющегося на одноразовое (`none`) — удаляются все дочерние вхождения. При изменении с одноразового на повторяющееся — генерируются дочерние вхождения на 60 дней.

#### Scenario: Изменение weekly на monthly
- **WHEN** владелец меняет `recurrence_rule` с `weekly` на `monthly`
- **THEN** все будущие дочерние вхождения удаляются, генерируются новые по правилу `monthly`

#### Scenario: Отключение повторения
- **WHEN** владелец меняет `recurrence_rule` на `none`
- **THEN** все дочерние вхождения удаляются, событие становится одноразовым

#### Scenario: Включение повторения
- **WHEN** владелец добавляет `recurrence_rule` к одноразовому событию
- **THEN** генерируются дочерние вхождения на 60 дней вперёд

### Requirement: Reminder recreation
При изменении date/time ДОЛЖНЫ удаляться неотправленные напоминания и создаваться новые по обновлённым данным, используя `reminder_settings` участников. Для повторяющихся событий — пересоздать напоминания для всех будущих вхождений.

#### Scenario: Пересоздание напоминаний при изменении времени серии
- **WHEN** владелец меняет `event_time` повторяющегося события
- **THEN** неотправленные напоминания удаляются у родителя и всех будущих вхождений, создаются новые по обновлённому времени
