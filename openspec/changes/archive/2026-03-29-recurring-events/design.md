## Context

Сейчас все события в системе одноразовые. Таблица `events` содержит `event_date` (DATE) и `event_time` (TIME nullable). Напоминания создаются в момент публикации события как записи `scheduled_reminders` с предвычисленным `remind_at`. Планировщик каждые 30 секунд проверяет `WHERE sent = FALSE AND remind_at <= NOW()`.

Пользователи запрашивают повторяющиеся события: еженедельные встречи, ежемесячные собрания, ежегодные дни рождения. Макет Mini App уже содержит UI для повторений (RepeatPicker с 6 опциями).

## Goals / Non-Goals

**Goals:**
- Поддержка 6 типов повторения: daily, weekly, biweekly, monthly, yearly
- Распознавание повторений из текста через LLM-парсер
- Выбор повторения в Mini App при создании и редактировании
- Автоматическая генерация будущих вхождений (occurrences) с напоминаниями
- Удаление/редактирование серии целиком
- Сохранение обратной совместимости (существующие события не затрагиваются)

**Non-Goals:**
- Кастомные правила повторения (каждые N дней, конкретные дни недели — только фиксированные 6 опций)
- iCalendar RRULE-формат (избыточно для текущих нужд)
- Редактирование отдельного вхождения (только серия целиком)
- Исключение дат из серии (exceptions)
- Бесконечные серии без конечной даты (используем rolling window)

## Decisions

### 1. Хранение: поле recurrence_rule в таблице events (а не отдельная таблица)

Правило повторения хранится как `VARCHAR(20)` в колонке `recurrence_rule` таблицы `events`. Допустимые значения: `daily`, `weekly`, `biweekly`, `monthly`, `yearly`, `NULL` (одноразовое).

**Альтернатива**: Отдельная таблица `recurrence_rules` с FK. Избыточно — у нас фиксированный набор из 5 вариантов без кастомных параметров.

### 2. Генерация вхождений: materialized occurrences (а не виртуальные)

Каждое вхождение повторяющегося события — отдельная строка в `events` со ссылкой на родительское событие через `parent_event_id`. Родительское событие (первое в серии) имеет `parent_event_id = NULL` и `recurrence_rule != NULL`.

**Почему materialized:**
- Напоминания (`scheduled_reminders`) уже привязаны к конкретному `event_id` — не нужно менять логику планировщика
- Конфликт-детекция работает без изменений (ищет по event_date/event_time)
- GET /events возвращает конкретные записи без runtime-вычислений
- Пользователи видят конкретные даты в списке событий

**Альтернатива**: Виртуальная генерация при запросе. Сложнее: нужно менять все запросы, планировщик напоминаний, конфликт-детекцию.

### 3. Rolling window: генерация на 60 дней вперёд

Фоновая задача (в том же APScheduler) генерирует вхождения на 60 дней вперёд от текущей даты. Запускается раз в сутки (или при создании/редактировании события). Не создаёт дубликаты — проверяет существование по `parent_event_id + occurrence_date`.

**Почему 60 дней**: Достаточно для всех типов повторения (monthly/yearly получат 1-2 вхождения). Ежедневные — до 60 записей, допустимо.

### 4. Дочерние вхождения: минимальная структура

Дочернее вхождение (`parent_event_id IS NOT NULL`):
- Наследует `title`, `event_time`, `space_id`, `created_by` от родителя
- Имеет собственные `event_date` (вычисленная дата вхождения) и `id`
- `recurrence_rule = NULL` (правило хранится только у родителя)
- Напоминания создаются для каждого вхождения индивидуально

### 5. Удаление серии: каскадное через parent_event_id

Удаление родительского события удаляет все дочерние вхождения (ON DELETE CASCADE). Удаление одного вхождения недоступно (non-goal).

При удалении через бот/Mini App: если событие имеет `recurrence_rule != NULL`, предупреждаем "Будут удалены все будущие повторения".

### 6. Редактирование серии

Изменение title/time родительского события обновляет все будущие дочерние вхождения (UPDATE WHERE parent_event_id = :id AND event_date >= CURRENT_DATE). Напоминания пересоздаются для обновлённых вхождений.

Изменение recurrence_rule: удаляем все будущие дочерние вхождения, генерируем новые по новому правилу.

### 7. LLM-парсер: новое поле recurrence_rule

Добавляем в JSON-ответ парсера поле `"recurrence_rule": "weekly" | "daily" | "biweekly" | "monthly" | "yearly" | null`. Расширяем промпт примерами: "каждую неделю", "ежедневно", "раз в месяц", "каждый год", "каждые две недели".

### 8. Вычисление следующей даты

| Правило | Логика |
|---------|--------|
| daily | +1 день |
| weekly | +7 дней |
| biweekly | +14 дней |
| monthly | +1 месяц (dateutil.relativedelta), если день > кол-ва дней в месяце — последний день |
| yearly | +1 год, 29 февр → 28 февр в не-високосный год |

Используем `dateutil.relativedelta` для monthly/yearly (уже в зависимостях через python-dateutil).

## Risks / Trade-offs

**[Рост таблицы events]** → Ежедневные события создают до 60 записей на 60 дней. При 100 ежедневных повторяющихся событиях — 6000 строк, приемлемо для PostgreSQL. Индекс `idx_events_space_date` покрывает типичные запросы.

**[Orphaned occurrences]** → Если генератор упадёт, старые вхождения останутся. Генератор идемпотентен — при следующем запуске создаст недостающие. Прошедшие вхождения не удаляются (они в истории).

**[Monthly edge cases]** → 31 января → monthly → 28/29 февраля. Используем relativedelta, который корректно обрабатывает эти случаи.

**[Конфликт при массовой генерации напоминаний]** → Генерация 60 вхождений × N участников × M интервалов = потенциально много INSERT. Решение: batch insert, генерация в отдельной транзакции с SAVEPOINT.

## Migration Plan

1. Alembic миграция: добавить `recurrence_rule` (VARCHAR 20, nullable) и `parent_event_id` (UUID, nullable FK → events.id ON DELETE CASCADE) в таблицу events
2. Индекс на `parent_event_id` для каскадных операций
3. Деплой кода с новой логикой (обратно совместим — все существующие события имеют NULL в новых полях)
4. Откат: миграция alembic downgrade (удаление колонок), потеря данных о повторениях минимальна (фича новая)
