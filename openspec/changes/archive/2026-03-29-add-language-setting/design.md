## Context

Все строки интерфейса (бот + Mini App) захардкожены на русском. Модель User не содержит поля языка. Telegram передаёт `language_code` пользователя в каждом сообщении — его можно использовать для автоопределения. Макет в `docs/shared-plan-miniapp.jsx` уже содержит готовую структуру `I18N` словаря с ключами `ru` и `en`, а также экран выбора языка.

## Goals / Non-Goals

**Goals:**
- Добавить поле `language` в модель User (хранение выбранного языка)
- Автоопределять язык при первом контакте из `language_code` Telegram (fallback — `en`)
- Перевести все строки бота (хендлеры, форматирование, клавиатуры, callbacks, middleware) на систему словарей
- Перевести все строки Mini App frontend на систему i18n-словарей
- Добавить API для чтения/обновления языка
- Добавить экран выбора языка в Mini App (по макету)

**Non-Goals:**
- Локализация LLM-промптов (парсинг событий остаётся на русском — LLM и так мультиязычный)
- Поддержка языков кроме ru/en
- Локализация Telegram-уведомлений (push) — они следуют языку получателя, т.к. уже привязаны к пользователю
- Изменение формата дат/чисел (остаётся как есть)

## Decisions

### 1. Хранение языка — поле `language` в таблице `users`

**Решение:** VARCHAR(5), NOT NULL, DEFAULT 'en'. Обновляется при первом контакте из `language_code`.

**Альтернатива:** Хранить в `reminder_settings` JSONB. Отклонено — язык не связан с напоминаниями, это отдельная пользовательская настройка. Отдельное поле проще для запросов и миграций.

### 2. Автоопределение при первом контакте

**Решение:** В `UserProfileMiddleware` при upsert: если пользователь новый, берём `language_code` из Telegram. Если `language_code` начинается с `ru` — ставим `ru`, иначе — `en`. Для существующих пользователей поле не перезаписывается (пользователь мог выбрать вручную).

**Альтернатива:** Определять при каждом запросе. Отклонено — пользователь мог сменить язык вручную, не хотим затирать его выбор.

### 3. Архитектура i18n для бота (Python)

**Решение:** Модуль `app/i18n.py` с двумя словарями (`MESSAGES_RU`, `MESSAGES_EN`) и функцией `t(lang, key, **kwargs)`. Все строки из `formatting.py`, `keyboards/`, `handlers/`, `callbacks/`, `middlewares/access_control.py` переносятся в словари. Ключи — плоские строки (напр. `"event.confirm.yes"`, `"reminder.label.1d"`).

**Альтернатива:** Использовать gettext / Babel. Отклонено — для 2 языков это overkill, простые словари проще поддерживать и не требуют дополнительных зависимостей.

### 4. Архитектура i18n для Mini App (React)

**Решение:** Файл `mini_app/frontend/src/i18n.ts` с объектом переводов (структура аналогична макету `I18N` из `docs/shared-plan-miniapp.jsx`). React-хук `useTranslation()` возвращает `t` (словарь текущего языка) и `setLanguage(code)`. Язык загружается с бэкенда при инициализации (`GET /api/user/language`).

**Альтернатива:** i18next. Отклонено — для 2 языков и ~80 ключей это overkill, плюс дополнительная зависимость.

### 5. Передача языка в хендлеры бота

**Решение:** `UserProfileMiddleware` уже делает upsert при каждом сообщении. После upsert достаём `language` из записи User и передаём в `data["lang"]`. Хендлеры получают `lang` из `data` и используют `t(lang, key)`.

### 6. API-эндпоинт для языка

**Решение:** Добавить в user-роуты Mini App:
- `GET /api/user/language` → `{"language": "ru"}`
- `PUT /api/user/language` (тело `{"language": "ru"}`) → обновляет, возвращает `{"language": "ru"}`

Валидация: `language` ДОЛЖЕН быть `"ru"` или `"en"`. Иначе — 400.

### 7. Экран выбора языка в Mini App

**Решение:** Новый маршрут `/settings/language`. Section с заголовком "Язык интерфейса" / "Interface language". Два варианта: "Русский" и "English". Выбранный — жирный шрифт + галочка. При выборе — PUT на API + немедленное переключение UI. Доступ с экрана пространств (блок "Настройки", рядом с "Напоминания"). Реализация по макету из `docs/shared-plan-miniapp.jsx`.

## Risks / Trade-offs

**[Большой объём изменений]** → Изменения затрагивают почти все файлы с UI-строками. Миграция рефакторинг-heavy, но каждый файл меняется механически (замена строки на вызов `t()`). Риск снижается тем, что можно мигрировать по файлам.

**[Рассинхронизация словарей]** → При добавлении новых строк можно забыть добавить перевод. Митигация: TypeScript-типы для ключей во фронте, runtime-проверка в Python (`t()` возвращает ключ как fallback при отсутствии перевода).

**[Язык уведомлений]** → Уведомления (напоминания, оповещения о событиях) отправляются конкретному пользователю — берём его `language` из БД. Каждый получает уведомление на своём языке.

**[Миграция существующих пользователей]** → Существующие пользователи получат `language = 'en'` (default). При следующем взаимодействии с ботом `UserProfileMiddleware` НЕ перезаписывает язык (пользователь уже "существующий"). Можно добавить одноразовую миграцию данных с `language_code`, но это non-goal для первой итерации. Пользователи смогут переключить язык вручную.
