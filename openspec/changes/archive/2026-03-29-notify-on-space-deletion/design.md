## Context

При удалении пространства через Mini App API (`DELETE /api/spaces/{space_id}`) каскадное удаление в PostgreSQL автоматически удаляет все связанные записи: `user_spaces`, `events`, `scheduled_reminders`. Участники не получают никакого уведомления — события и напоминания молча исчезают.

Существует устоявшийся паттерн уведомления участников: функция `_notify_space_members()` в `mini_app/backend/routes/events.py` уже реализует цикл по участникам с отправкой через `bot.send_message()` и error isolation (try/except на каждого).

## Goals / Non-Goals

**Goals:**
- Уведомить всех участников пространства (кроме инициатора удаления) о том, что пространство удалено
- Сохранить данные пространства и список участников до каскадного удаления
- Следовать существующему паттерну уведомлений (error isolation, logging)

**Non-Goals:**
- Уведомление при удалении через другие каналы (прямой SQL, бот-команды — их нет)
- Подтверждение доставки уведомлений или повторные попытки
- Уведомление о выгнании (kick) отдельного участника — отдельная задача

## Decisions

### 1. Порядок операций: сначала собрать данные, потом удалить, потом уведомить

Каскадное удаление уничтожает `user_spaces` — список участников нужно получить **до** удаления. Однако уведомления отправляются **после** удаления, чтобы не блокировать API-ответ в случае ошибки отправки и чтобы не уведомлять, если удаление по какой-то причине не прошло.

Порядок:
1. Получить `space.name` и список участников (`get_space_members`)
2. Получить `first_name` инициатора (из User по `user_id`)
3. Вызвать `delete_space()` + `session.commit()`
4. Отправить уведомления (fire-and-forget, ошибки логируются)
5. Вернуть HTTP 204

**Альтернатива:** отправлять уведомления до удаления. Отклонено — если удаление потом упадёт, пользователи получат ложное уведомление.

### 2. Переиспользовать `_notify_space_members` из events.py

В `mini_app/backend/routes/events.py` уже есть helper `_notify_space_members(request, session, space_id, exclude_user_id, message)`. Однако он запрашивает участников из `user_spaces`, которые к моменту вызова будут удалены каскадом. Поэтому нельзя использовать его напрямую.

Решение: написать inline-логику отправки уведомлений в `delete_space` route, аналогично паттерну в `events.py`, но используя заранее полученный список участников.

### 3. Форматирование уведомления — новая функция в `formatting.py`

Добавить `format_space_deleted_notification(space_name, admin_name)` в `app/bot/formatting.py`. Формат уведомления:

```
🗑 Пространство «{space_name}» удалено

👤 Удалил: {admin_name}
ℹ️ Напоминания из этого пространства больше не будут приходить.
```

Стилистически согласуется с `format_event_deleted_notification`.

### 4. Ручной commit перед уведомлениями

Middleware `db_session_middleware` делает commit автоматически после handler. Но нам нужно, чтобы удаление было зафиксировано **до** отправки уведомлений. Решение: вызвать `await session.commit()` явно после `delete_space()`, затем отправить уведомления, затем вернуть response. Повторный commit middleware будет no-op.

## Risks / Trade-offs

- **[Race condition]** Если между получением списка участников и удалением кто-то покинет пространство, он получит лишнее уведомление → Допустимо, уведомление безвредно.
- **[Ошибка отправки]** Если бот недоступен или Telegram API упал, часть участников не получит уведомление → Допустимо по паттерну error isolation. Логируем и продолжаем. Удаление уже произошло.
- **[Медленная отправка]** При большом количестве участников sequential-отправка может замедлить ответ API → В контексте проекта (небольшие группы) это несущественно.
