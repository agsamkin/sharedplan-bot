## Context

Бот уже поддерживает полный text-to-event пайплайн (Фаза 4): текстовое сообщение → LLM-парсинг через OpenRouter → карточка подтверждения → сохранение → уведомления. Обработчик `event.py` содержит логику выбора пространства и отображения карточки, привязанную к текстовому вводу.

Голосовые сообщения Telegram приходят в формате OGG/Opus. Nexara API поддерживает этот формат нативно — конвертация не требуется. API совместим с OpenAI Whisper по интерфейсу, но мы используем `aiohttp` для async multipart upload (как указано в архитектуре).

Конфигурация `NEXARA_API_KEY` уже определена в `config.py`.

## Goals / Non-Goals

**Goals:**
- Добавить voice-to-event пайплайн: голосовое → транскрипция → LLM-парсинг → подтверждение
- Переиспользовать существующий event flow (выбор пространства, подтверждение, сохранение)
- Показывать транскрипцию в карточке подтверждения для верификации пользователем
- Обеспечить изоляцию ошибок: сбой STT не влияет на текстовый флоу

**Non-Goals:**
- Поддержка видеосообщений (только `Message.voice`)
- Тюнинг распознавания (язык, модель) — используем настройки по умолчанию Nexara
- Кэширование транскрипций
- Поддержка диаризации (определение говорящих)

## Decisions

### 1. Клиент Nexara: aiohttp с multipart upload

**Решение:** Использовать `aiohttp.ClientSession` для async POST с `multipart/form-data`.

**Альтернативы:**
- OpenAI SDK с `base_url` — Nexara совместим, но SDK синхронный по умолчанию, а async-клиент openai уже используется для OpenRouter. Два клиента с разными base_url добавят путаницу.
- httpx — ещё одна зависимость. aiohttp уже в проекте.

**Обоснование:** aiohttp уже в зависимостях, multipart upload тривиален. Изолированный клиент `speech_to_text.py` без дополнительных зависимостей.

### 2. Скачивание аудио в память (BytesIO), не на диск

**Решение:** `bot.download(file_id)` → `BytesIO` → передача байтов в Nexara.

**Обоснование:** Голосовые сообщения Telegram малы (обычно < 1 МБ). Запись на диск контейнера создаёт необходимость очистки и потенциальные проблемы с правами. In-memory проще и быстрее.

### 3. Рефакторинг общей логики event flow

**Решение:** Извлечь из `event.py` общую функцию `_process_parsed_event(message, state, session, bot, parsed, raw_input, transcript=None)`, которую вызывают оба обработчика (текст и голос).

**Альтернативы:**
- Дублировать логику в voice.py — нарушает DRY, усложняет поддержку.
- Вынести в отдельный модуль — overhead для двух вызовов.

**Обоснование:** Функция в `event.py` — минимальное изменение. Voice handler импортирует и вызывает её, передавая дополнительный параметр `transcript` для отображения в карточке.

### 4. Карточка подтверждения с транскрипцией

**Решение:** При наличии `transcript` в FSM-данных, карточка подтверждения дополняется строкой `🎤 Распознано: «{transcript}»` перед данными события.

**Обоснование:** Пользователь должен видеть, что именно было распознано из голоса, чтобы поймать ошибки STT до подтверждения. Для текстового ввода транскрипция не показывается.

### 5. raw_input для голосовых сообщений

**Решение:** В `events.raw_input` сохраняется транскрибированный текст (не аудиобайты). Это позволяет отлаживать цепочку STT → LLM.

**Обоснование:** Колонка `raw_input` уже существует и используется для текстовых событий. Для голосовых она хранит транскрипцию — тот текст, который пошёл в LLM-парсер.

## Risks / Trade-offs

- **[Качество транскрипции]** → Nexara может некорректно распознать текст, особенно имена собственные и числа. Митигация: транскрипция отображается в карточке, пользователь может отредактировать или отменить.
- **[Латентность]** → Двойной API-вызов (STT + LLM) увеличивает задержку до 5–15 секунд. Митигация: `chat_action: typing` индицирует обработку. Это приемлемо для UX голосовых сообщений.
- **[Nexara недоступен]** → Понятное сообщение об ошибке + предложение написать текстом. Текстовый флоу продолжает работать независимо.
