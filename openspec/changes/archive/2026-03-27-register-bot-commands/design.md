## Context

Бот имеет 7 пользовательских команд (`/start`, `/help`, `/newspace`, `/spaces`, `/space_info`, `/events`, `/reminders`), но не вызывает `bot.set_my_commands()` при старте. Telegram не показывает список команд в меню (кнопка «/»).

## Goals / Non-Goals

**Goals:**
- Регистрировать команды бота через Telegram Bot API при каждом старте
- Определить список команд в одном месте для простоты поддержки

**Non-Goals:**
- Разные меню для разных ролей (admin vs member) — через `BotCommandScope`
- Локализация описаний команд на несколько языков
- Динамическое обновление меню в runtime

## Decisions

### Список команд определяется как константа в `app/bot/commands.py`

Список `BotCommand` объектов хранится в отдельном модуле-константе. Это позволяет импортировать его из `main.py` для регистрации, а в будущем — из тестов или других модулей.

Альтернатива — определить список inline в `main.py`. Отклонено: при росте числа команд `main.py` раздуется, и список сложнее переиспользовать.

### Вызов `set_my_commands` в `main()` после создания `Bot`

Вызов размещается в функции `main()` в `app/main.py`, после `bot.get_me()` и до `dp.start_polling()`. Это гарантирует, что команды регистрируются при каждом старте бота.

Альтернатива — startup hook через `dp.startup.register()`. Отклонено: явный вызов проще и прозрачнее для такой простой операции.

### Scope по умолчанию (BotCommandScopeDefault)

Команды регистрируются для всех пользователей без разделения по scope. Для MVP этого достаточно.

## Risks / Trade-offs

- **Rate limit при частых рестартах** → Маловероятно: `set_my_commands` вызывается один раз при старте. Telegram rate limit для этого метода щадящий.
- **Рассинхронизация списка команд с хендлерами** → Минимизировано: список определён в одном файле, легко сверять вручную. В будущем можно добавить автоматическую проверку.
