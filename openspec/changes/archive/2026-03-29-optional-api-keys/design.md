## Context

Сейчас `app/config.py` определяет `OPENROUTER_API_KEY`, `OPENROUTER_MODEL` и `NEXARA_API_KEY` как обязательные поля pydantic-settings. При их отсутствии приложение падает до старта. Это делает невозможным использование бота без подключения к внешним сервисам, хотя Mini App позволяет создавать события вручную.

Ключевые точки использования:
- `app/services/speech_to_text.py` — `settings.NEXARA_API_KEY` в заголовке Authorization
- `app/services/llm_parser.py` — `settings.OPENROUTER_API_KEY` при ленивой инициализации клиента AsyncOpenAI
- `app/bot/handlers/voice.py` — вызывает `transcribe()` → `parse_event()`
- `app/bot/handlers/event.py` — вызывает `parse_event()`

## Goals / Non-Goals

**Goals:**
- Бот стартует и функционирует без ключей Nexara и OpenRouter
- Пользователь получает понятное сообщение при попытке использовать отключённую функцию
- Отключённые функции логируются при старте
- Mini App работает полностью независимо от наличия ключей

**Non-Goals:**
- Fallback на альтернативные STT/LLM провайдеры
- Автоматическое переключение между режимами при изменении конфигурации (требует перезапуск)
- UI-индикация отключённых функций в Mini App (там ручной ввод всегда доступен)

## Decisions

### 1. Проверка доступности на уровне сервисов, а не хендлеров

Функции `transcribe()` и `parse_event()` сами проверяют наличие ключа и выбрасывают ошибку с типом `service_disabled`. Хендлеры обрабатывают этот тип как обычную ошибку через существующий маппинг сообщений.

**Альтернатива**: проверять в хендлерах перед вызовом сервиса. Отвергнута — дублирование логики проверки, нарушение инкапсуляции сервисов.

### 2. Optional поля с default None в Settings

`OPENROUTER_API_KEY`, `OPENROUTER_MODEL`, `NEXARA_API_KEY` становятся `Optional[str] = None`. При `OPENROUTER_API_KEY=None` оба поля OpenRouter считаются недоступными.

**Альтернатива**: отдельные boolean-флаги `ENABLE_STT`, `ENABLE_LLM`. Отвергнута — избыточно, наличие/отсутствие ключа — достаточный сигнал.

### 3. Ранняя проверка в сервисных функциях

`transcribe()` проверяет `settings.NEXARA_API_KEY is None` первой строкой и выбрасывает `TranscriptionError(error_type="service_disabled")`. Аналогично `parse_event()` проверяет ключ и выбрасывает `ParseError(error_type="service_disabled")`.

Проверка ленивая (при вызове), не при импорте — консистентно с текущей ленивой инициализацией клиента в `llm_parser.py`.

### 4. Сообщения пользователю

- Голосовое сообщение без Nexara: «Функция голосовых сообщений отключена. Напиши текстом или создай событие в приложении.»
- Текстовое сообщение без OpenRouter: «Автоматическое распознавание событий отключено. Создай событие вручную в приложении.» + inline-кнопка для открытия Mini App (если `MINI_APP_URL` задан).
- Голосовое сообщение без OpenRouter (Nexara есть, но LLM нет): та же ошибка `service_disabled` от `parse_event()` — бот покажет транскрипцию и предложит Mini App.

### 5. Логирование при старте

В `app/main.py` после успешных проверок PostgreSQL и Telegram-токена добавляется блок логирования:
```
INFO: Функции: STT (Nexara) — ✓/✗, LLM-парсинг (OpenRouter) — ✓/✗
```

Это не блокирующая проверка — просто информация.

## Risks / Trade-offs

- **Пользователь может не понять, почему функция отключена** → Митигация: сообщение об ошибке прямо предлагает альтернативу (Mini App / текстовый ввод)
- **Голосовой ввод без LLM бесполезен** (транскрипция есть, но парсинг невозможен) → Митигация: покажем транскрипцию и предложим Mini App. Если и Nexara, и OpenRouter отключены — сообщаем сразу про отключённые голосовые сообщения
- **OPENROUTER_MODEL без OPENROUTER_API_KEY** → Игнорируется, модель не нужна без ключа
