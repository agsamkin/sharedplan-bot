## ADDED Requirements

### Requirement: Логирование доступности функций при старте

После успешного прохождения обязательных проверок (PostgreSQL, миграции, Telegram-токен) система ДОЛЖНА логировать информацию о доступности опциональных функций на основе наличия API-ключей.

#### Scenario: Все API-ключи заданы

- **WHEN** приложение стартует и `OPENROUTER_API_KEY` и `NEXARA_API_KEY` заданы
- **THEN** в лог записывается INFO: «Функции: STT (Nexara) — включён, LLM-парсинг (OpenRouter) — включён»

#### Scenario: Ключи отсутствуют

- **WHEN** приложение стартует и `OPENROUTER_API_KEY` и `NEXARA_API_KEY` не заданы
- **THEN** в лог записывается WARNING: «Функции: STT (Nexara) — отключён, LLM-парсинг (OpenRouter) — отключён»

#### Scenario: Частично заданы ключи

- **WHEN** приложение стартует с `NEXARA_API_KEY` заданным, но без `OPENROUTER_API_KEY`
- **THEN** в лог записывается информация о статусе каждой функции отдельно

## MODIFIED Requirements

### Requirement: Порядок проверок при старте

Проверки ДОЛЖНЫ выполняться в определённом порядке: (1) env-переменные (pydantic-settings, уже есть), (2) PostgreSQL connectivity, (3) Alembic миграции, (4) Telegram token, (5) логирование доступности функций. Если шаг N (1-4) завершился ошибкой, шаги N+1... не выполняются. Шаг 5 — информационный, не блокирующий.

#### Scenario: Все проверки прошли

- **WHEN** все env-переменные заданы, PostgreSQL доступен, миграции применены, токен валидный
- **THEN** логируется доступность функций, запускается APScheduler и long-polling

#### Scenario: Ошибка на шаге PostgreSQL

- **WHEN** env-переменные валидны, но PostgreSQL недоступен
- **THEN** миграции не запускаются, токен не проверяется, приложение завершается
