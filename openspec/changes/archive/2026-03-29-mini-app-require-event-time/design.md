## Context

В Mini App экран создания события (`EventCreatePage.tsx`) имеет три поля: название, дата, время. Время помечено как «необязательно», и кнопка «Создать событие» активна при заполнении только названия и даты. Backend API (`POST /api/spaces/:space_id/events`) принимает `event_time` как опциональное поле — это поведение нужно для событий, создаваемых через бота (текст/голос), где время может быть не указано.

## Goals / Non-Goals

**Goals:**
- Сделать время обязательным полем в форме Mini App
- Кнопка «Создать событие» активна только при заполнении всех трёх полей

**Non-Goals:**
- Изменение backend API (event_time остаётся опциональным для бота)
- Изменение поведения бота при парсинге событий без времени
- Добавление валидации времени на стороне backend для Mini App

## Decisions

**Решение 1: Только frontend-валидация**

Изменения ограничены компонентом `EventCreatePage.tsx`. Backend API не меняется, чтобы не ломать создание событий через бота.

Альтернатива: добавить параметр `source=miniapp` в API и требовать время для этого источника — избыточно для текущей задачи.

**Решение 2: Удаление пометки «необязательно» вместо добавления «обязательно»**

Убираем `(необязательно)` из лейбла времени. Все три поля визуально выглядят одинаково — без дополнительных пометок. Это проще и чище, чем добавлять «обязательно» к каждому полю.

## Risks / Trade-offs

**[Расхождение frontend/backend]** → Frontend требует время, backend — нет. Если кто-то отправит запрос напрямую к API без времени, событие создастся. Это приемлемо: API защищён авторизацией Telegram initData, прямой доступ маловероятен.

**[Сценарий «событие без конкретного времени»]** → Пользователь Mini App больше не может создать событие без времени (например, «весь день»). Это осознанный trade-off: для совместного планирования время важнее гибкости. Через бота такие события по-прежнему можно создать.
