## Context

Бот полностью функционален после Фаз 1–6: пространства, события, AI-парсинг, голос, напоминания. Однако ряд защитных механизмов отсутствует: нет валидации длины текста события, нет проверки даты на прошедшую/будущую, first_name не обновляется вне `/start`, при старте нет проверки связности с PostgreSQL и валидности токена.

Логирование есть, но неоднородно: LLM и STT логируются детально, а пользовательские команды — нет. Формат логов не стандартизирован.

Текущее состояние error handling:
- **LLM (llm_parser.py)**: retry 3x с backoff, ParseError с типами, логирование — всё на месте
- **STT (speech_to_text.py)**: retry 2x, TranscriptionError, логирование — на месте
- **Handlers**: ParseError/TranscriptionError ловятся, пользователь получает сообщение через `formatting.py`
- **Notifications**: try/except в start.py, event_confirm.py, jobs.py — на месте
- **DB middleware**: commit/rollback — на месте

## Goals / Non-Goals

**Goals:**
- Валидация пользовательского ввода: длина текста события (1–1000 символов), диапазон даты (не более 2 лет в будущее), предупреждение о прошедшей дате
- Обновление first_name/username при каждом взаимодействии (middleware)
- Проверки при старте: PostgreSQL connectivity, Telegram token (getMe), env-переменные
- Стандартизация логирования: единый формат, логирование всех команд

**Non-Goals:**
- Rate limiting (личный бот)
- Sentry или внешний мониторинг
- Нагрузочное тестирование
- Юнит/интеграционные тесты
- Retry для неотправленных напоминаний (sent=TRUE при ошибке — осознанное решение)

## Decisions

### 1. Middleware для обновления профиля пользователя

**Решение**: Новый middleware `UserProfileMiddleware` обновляет `first_name` и `username` при каждом входящем сообщении/callback.

**Альтернатива**: Обновлять в каждом handler вручную — дублирование кода, легко забыть.

**Реализация**: Middleware вызывает upsert через `space_service.get_or_create_user()` (уже существующий метод) перед передачей управления handler'у. Middleware регистрируется после `DbSessionMiddleware` (нужна сессия).

### 2. Валидация ввода в handlers, не в services

**Решение**: Проверка длины текста и диапазона даты выполняется в handler'ах (event.py, voice.py, callbacks), а не в service-слое.

**Альтернатива**: Валидация в service — логичнее по архитектуре, но validation feedback напрямую зависит от Telegram (отправка сообщения, inline-кнопки), что нарушает разделение слоёв.

**Компромисс**: Валидация диапазона даты в ParsedEvent Pydantic-модели (2 года), предупреждение о прошедшей дате — в handler.

### 3. Предупреждение о прошедшей дате через inline-кнопки

**Решение**: Если LLM вернул дату в прошлом, показываем предупреждение с двумя кнопками: `[✅ Всё равно создать]` и `[❌ Отмена]`. При подтверждении — стандартный флоу.

**Реализация**: В handler после парсинга проверяем `event_date < today`. Если да — отправляем предупреждающую карточку вместо стандартной. Callback `event_past_confirm:{uuid}` переводит в стандартный флоу подтверждения.

### 4. Startup checks — fail-fast при старте

**Решение**: Перед запуском polling выполняются последовательные проверки:
1. pydantic-settings валидирует env (уже есть)
2. `SELECT 1` к PostgreSQL через async engine
3. `bot.get_me()` для проверки токена

При ошибке любой проверки — `sys.exit(1)` с описательным сообщением.

**Альтернатива**: Продолжать запуск и падать позже — хуже для отладки, запутанные ошибки.

### 5. Стандартизация логирования — без внешних библиотек

**Решение**: Стандартный `logging` с единым форматом. Добавляем логирование команд через middleware или в начале каждого handler'а.

**Формат**: `"%(asctime)s [%(levelname)s] %(name)s: %(message)s"` — простой, читаемый.

**Логирование команд**: В каждом handler команды — `logger.info("/%s user_id=%d", command, user_id)`. Не через middleware, потому что middleware не знает семантику команды (space_id, action).

**Альтернатива**: structlog / JSON-логирование — overkill для личного бота.

### 6. Валидация даты в ParsedEvent

**Решение**: Добавляем Pydantic-валидатор в `ParsedEvent`, который проверяет: дата не более 2 лет в будущее. Если дата выходит за диапазон — `ParseError(invalid_json)`.

**Прошедшие даты**: НЕ блокируем на уровне Pydantic — пользователь может сознательно хотеть создать событие в прошлом (запись задним числом). Предупреждение — в handler.

## Risks / Trade-offs

**[Middleware обновления профиля добавляет DB-запрос на каждое взаимодействие]** → Минимальный overhead: один upsert-запрос. Для личного бота с ~50 пользователями — незаметно. `get_or_create_user` уже использует `ON CONFLICT DO UPDATE`, что эффективно.

**[Предупреждение о прошедшей дате усложняет FSM-флоу]** → Флоу линейный: парсинг → проверка → предупреждение ИЛИ подтверждение. Не создаёт новых FSM-состояний — используем inline-кнопки с callback data, данные события хранятся в FSM state.

**[Startup checks блокируют запуск]** → Именно это и нужно. Лучше не запуститься с понятной ошибкой, чем запуститься и падать на каждом запросе.
