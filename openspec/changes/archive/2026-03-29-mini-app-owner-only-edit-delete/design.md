## Context

В Mini App экран EventDetailPage (`/events/:id`) сейчас отображает полную форму редактирования для всех участников пространства: input-поля для названия, даты, времени, кнопку «Сохранить» и кнопку «Удалить событие». Бэкенд уже корректно проверяет владение (поле `created_by` в таблице `events`) и возвращает 403 при попытке редактирования/удаления чужого события. Также бэкенд уже возвращает поле `is_owner: boolean` в ответе `GET /api/events/:id`.

Для пространств аналогичная проблема уже решена: `SpaceDetailPage.tsx` вычисляет `isAdmin` через сравнение `currentUserId` с `space.created_by` и скрывает кнопки «Изменить» и «Удалить» для не-администраторов.

Задача — сделать то же самое для событий на экране EventDetailPage, используя уже существующее поле `is_owner` из API.

## Goals / Non-Goals

**Goals:**
- Для не-владельца: поля формы события отображаются в read-only режиме (информация видна, редактирование заблокировано)
- Для не-владельца: кнопки «Сохранить» и «Удалить событие» скрыты
- Для владельца: поведение не меняется (полная форма редактирования)
- Использовать существующее поле `is_owner` из ответа API

**Non-Goals:**
- Изменения в бэкенд API (уже работает корректно)
- Изменение логики владения или добавление «co-owner» роли
- Возможность передачи владения событием
- Изменения в SpaceDetailPage (уже работает корректно)
- Изменения в экране создания события (EventCreatePage)

## Decisions

### 1. Источник данных о владении — поле `is_owner` из API

**Решение**: использовать поле `is_owner`, которое уже возвращается бэкендом в `GET /api/events/:id`. Сохранять его в state компонента при загрузке события.

**Альтернатива**: вычислять владение на фронте через `Telegram.WebApp.initDataUnsafe.user.id === event.created_by`, аналогично SpaceDetailPage. Отклонена: дублирование логики, `is_owner` уже вычисляется на бэкенде и более надёжен (не зависит от доступности `initDataUnsafe`).

### 2. Read-only режим — через атрибут `readOnly` на input-элементах

**Решение**: для не-владельца добавить `readOnly` на все input-поля (название, дата, время) и визуально приглушить их (убрать border, уменьшить opacity). Это позволяет пользователю видеть данные события, но не редактировать.

**Альтернатива**: заменять input-ы на текстовые элементы (`<span>`) для не-владельца. Отклонена: значительно больше кода, дублирование layout, при этом `readOnly` даёт тот же результат с минимальными изменениями.

### 3. Условное скрытие кнопок — через проверку `isOwner`

**Решение**: обернуть блоки кнопок «Сохранить» и «Удалить событие» в `{isOwner && (...)}`. Для не-владельца кнопки полностью отсутствуют в DOM.

**Альтернатива**: показывать кнопки в disabled-состоянии с tooltip. Отклонена: кнопки без действия создают ложные ожидания, скрытие чище и проще.

### 4. Заголовок страницы зависит от владения

**Решение**: для владельца заголовок «Редактирование», для не-владельца — «Событие». Это явно сигнализирует о режиме просмотра.

## Risks / Trade-offs

- **Поле `is_owner` зависит от корректной аутентификации** → Аутентификация уже проверяется middleware на бэкенде (HMAC-SHA256 валидация initData). Если аутентификация сломается — все запросы вернут 401 до того, как дойдёт до проверки владения.
- **Пользователь может попытаться обойти read-only через DevTools** → Бэкенд уже проверяет владение и вернёт 403. Frontend-ограничения — только UX-улучшение, не security boundary.
- **Минимальный объём изменений** → Затрагивается только один файл (`EventDetailPage.tsx`), что снижает риск регрессий.
