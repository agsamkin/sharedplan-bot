## Context

Все 7 фаз реализованы. Бот функционально полный: создание пространств, события из текста/голоса, AI-парсинг через OpenRouter, напоминания через APScheduler, валидация и логирование. Фаза 8 — финальная полировка без изменения архитектуры или схемы БД.

Текущее состояние:
- Промпт парсера содержит базовые инструкции, но не имеет few-shot примеров для edge cases
- Форматирование дат ограничено: «сегодня», «завтра» и числовой формат — нет «послезавтра», нет дня недели
- Тексты сообщений функциональны, но не выверены на единообразие тона
- Карточка подтверждения не показывает день недели
- Напоминания не содержат дату события (только относительное время)

## Goals / Non-Goals

**Goals:**
- Повысить точность парсинга событий из естественного языка на типичных пользовательских запросах
- Сделать все пользовательские сообщения информативными, единообразными и дружелюбными
- Расширить форматирование дат для лучшей ориентации во времени
- Подготовить бота к комфортному ежедневному использованию

**Non-Goals:**
- Автоматическое тестирование промптов (нет test suite для LLM)
- Юнит/интеграционные тесты (личный проект)
- Оптимизация производительности
- Сравнение моделей OpenRouter (выходит за рамки кодовых изменений)
- Изменение архитектуры, схемы БД или API-контрактов

## Decisions

### 1. Few-shot примеры в промпте

Добавить 5-7 few-shot примеров в системный промпт парсера для покрытия основных edge cases: относительные даты, разговорные выражения времени, события без времени, события без явного названия.

**Почему few-shot, а не детальные инструкции:** LLM лучше обобщают из примеров, чем из абстрактных правил. Текущие текстовые инструкции по разрешению дат уже достаточны — few-shot дополнят их конкретикой.

**Альтернатива — chain-of-thought:** отклонена, т.к. увеличивает latency и стоимость, а задача парсинга достаточно простая.

### 2. Temperature остаётся 0.1

Текущее значение `temperature: 0.1` — хороший баланс. Значение 0.05 слишком детерминистично (LLM может «залипнуть» на неправильной интерпретации), 0.2 — слишком креативно для задачи извлечения структурированных данных.

### 3. Расширенное форматирование дат

Расширить `format_date_relative` в `formatting.py`:
- Добавить «послезавтра» для delta == 2
- Для дат в пределах текущей недели (delta 2-6) показывать день недели: «в среду», «в пятницу»
- Для более далёких дат оставить числовой формат с месяцем

**Почему не ISO-формат:** целевая аудитория — русскоязычные пользователи в неформальном контексте.

### 4. День недели в карточке подтверждения

Добавить день недели в формат даты карточки подтверждения: «27 марта 2026, пятница». Это помогает пользователю быстро верифицировать, что AI правильно интерпретировал относительную дату.

### 5. Дата события в напоминаниях

Добавить строку с датой в сообщение напоминания. Текущий формат показывает только относительное время («Через 15 минут»), но не абсолютную дату/время. Для напоминания «За 1 день» пользователю важно видеть конкретную дату.

### 6. Единый модуль сообщений не создаётся

Тексты сообщений остаются в месте использования (handlers, formatting.py, reminder_service.py). Централизация в отдельный модуль `messages.py` добавила бы indirection без реальной пользы — тексты не переиспользуются между компонентами, а при необходимости легко находятся grep-ом.

## Risks / Trade-offs

- **[Увеличение размера промпта]** → Few-shot примеры увеличат prompt на ~300 токенов. При стоимости ~$0.001 за запрос — незначительно. Latency увеличится на ~100-200ms — приемлемо.
- **[Регрессия парсинга]** → Изменение промпта может сломать работающие кейсы. Митигация: тестирование на реальных примерах до и после изменения (ручное, в рамках фазы).
- **[Форматирование дней недели]** → Русские склонения дней недели требуют винительного падежа («в среду», «в пятницу»). Нужна таблица склонений, а не просто именительный падеж.
