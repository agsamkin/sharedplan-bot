## Context

Сейчас события создаются исключительно через Telegram-бот (текст/голос → LLM-парсинг → подтверждение). Mini App позволяет просматривать и редактировать события, но не создавать новые. Экран EventDetail уже содержит форму с полями title/date/time — форма создания повторяет ту же структуру. Backend API имеет эндпоинты для GET/PUT/DELETE событий, но не POST. Сервисный слой `event_service` уже поддерживает `create_event()` и создание напоминаний.

## Goals / Non-Goals

**Goals:**
- Добавить экран создания события в Mini App с формой (название, дата, время)
- Добавить POST-эндпоинт для создания события через API
- Обеспечить создание напоминаний и уведомление участников при создании через Mini App
- Валидация: прошедшие даты (предупреждение), конфликты по времени (предупреждение), пустое название, длина > 500

**Non-Goals:**
- AI-парсинг текста в Mini App (остаётся только в боте)
- Голосовой ввод в Mini App
- Повторяющиеся события
- Выбор пространства при создании (контекст пространства уже задан — создание из экрана Space Detail)

## Decisions

### 1. POST-эндпоинт в контексте пространства
Эндпоинт: `POST /api/spaces/:space_id/events` (а не `POST /api/events`).

**Почему**: Событие всегда принадлежит пространству. URL соответствует REST-конвенции проекта (GET events уже под `/api/spaces/:space_id/events`). Проверка членства в пространстве — на уровне хендлера (как в существующих эндпоинтах).

**Альтернатива**: `POST /api/events` с `space_id` в теле — отклонено, т.к. нарушает существующую URL-иерархию.

### 2. Переиспользование event_service.create_event()
Бэкенд-хендлер вызывает тот же `event_service.create_event()`, что и бот-хендлер, затем `event_service.create_reminders_for_event()` и отправку уведомлений через бота.

**Почему**: Единая точка создания события — одинаковая логика напоминаний и уведомлений. Бот-объект уже доступен в API через `app['bot']`.

### 3. Навигация к форме создания с экрана Space Detail
Кнопка «Добавить» размещается в заголовке секции событий (рядом с надписью "СОБЫТИЯ * N"), а не в панели action-кнопок.

**Почему**: Соответствует макету. Кнопка контекстуально связана со списком событий, а не с управлением пространством.

### 4. Роут `/spaces/:id/events/new`
Новый роут для экрана создания. Формат вложенный, консистентный с `/spaces/:id/edit`.

**Почему**: Создание события привязано к конкретному пространству. Альтернатива `/events/new?spaceId=X` — менее чистая для React Router.

### 5. Предупреждения о прошедшей дате и конфликтах — на фронтенде
Валидация прошедшей даты выполняется на фронтенде (сравнение с текущей датой). Проверка конфликтов — через API-эндпоинт (тот же список событий пространства) на фронтенде.

**Почему**: Бэкенд не блокирует создание по этим причинам (как и в боте). Предупреждения — информативные, а не блокирующие. Данные о событиях уже загружены на экране Space Detail.

## Risks / Trade-offs

- **[Дублирование логики уведомлений]** → Уведомления при создании через API аналогичны уведомлениям из бота. Митигация: переиспользовать `notify_space_members` из event_service, а не дублировать код в хендлере.
- **[raw_input для событий из Mini App]** → При создании через бот сохраняется `raw_input` (исходный текст). При создании через Mini App `raw_input` будет пустым или содержать title. Митигация: `raw_input` — nullable поле, передаём `None`.
- **[Предупреждение о конфликтах на фронтенде]** → Между загрузкой списка событий и отправкой формы могут появиться новые события. Митигация: риск минимален для маленьких групп; добавление серверной проверки — overengineering для текущего масштаба.
