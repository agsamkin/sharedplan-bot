## Context

Системный промпт в `app/prompts/event_parser.py` содержит few-shot примеры с хардкоженной датой `2026-03-27 (пятница)`. При этом актуальная дата передаётся через `{current_date}` в начале промпта. Когда реальная дата отличается от даты в примерах (а она отличается всегда, кроме одного дня), LLM путается: для "завтра" может вычислить дату от примера (2026-03-28), а не от актуальной даты.

Текущая структура: `SYSTEM_PROMPT` — единая f-строка с плейсхолдерами `{current_date}`, `{current_time}`, `{weekday}`. Примеры вшиты в строку как литералы.

## Goals / Non-Goals

**Goals:**
- Устранить конфликт между датами в примерах и актуальной датой
- Сделать few-shot примеры динамическими, вычисляемыми от текущей даты
- Усилить инструкцию о приоритете актуальной даты
- Добавить граничные примеры (конец месяца, конец года, воскресенье → понедельник)

**Non-Goals:**
- Менять формат ответа LLM (JSON-схема остаётся прежней)
- Менять модель или провайдера (OpenRouter)
- Добавлять пост-валидацию дат на уровне кода (на данный момент это не стоит усложнения — сначала фиксим промпт)
- Менять retry-логику или параметры запроса

## Decisions

### 1. Динамическая генерация примеров вместо статических строк

**Выбор:** Функция `build_examples(current_date)`, которая вычисляет все даты в примерах относительно `current_date`.

**Альтернатива: Убрать примеры совсем, оставить только правила.** Отклонено — few-shot примеры критически важны для качества парсинга, особенно для граничных случаев (дефис как время, диапазоны, разговорные выражения).

**Альтернатива: Добавить в промпт "Игнорируй даты в примерах".** Отклонено — это контринтуитивно для LLM; проще дать консистентные данные, чем просить игнорировать часть контекста.

**Реализация:**
- Примеры определяются как список структур `(input_text, title, date_offset_days, time_or_none)`
- `date_offset_days` — смещение от `current_date` (1 = завтра, 2 = послезавтра)
- Для дней недели — функция-хелпер `next_weekday(current_date, target_weekday)`
- Для абсолютных дат (5 апреля, Новый год) — дата вычисляется напрямую, без offset
- Результирующая строка примеров вставляется в системный промпт

### 2. Разделение промпта на структурные блоки

**Выбор:** Разбить `SYSTEM_PROMPT` на: `PROMPT_HEADER` (роль + текущая дата + правила) + динамические примеры + `PROMPT_FOOTER` (финальное напоминание).

Это чище, чем одна строка со вставкой. Функция `build_system_prompt(current_date, current_time)` собирает финальный промпт.

### 3. Усиление инструкции о приоритете даты

**Выбор:** Добавить в начало промпта (после строки с текущей датой) явное правило:

> "КРИТИЧЕСКИ ВАЖНО: Все относительные даты (завтра, в понедельник, через 3 дня) вычисляй ТОЛЬКО от указанной выше даты «Сегодня». Примеры ниже приведены для этой же даты."

И в конце промпта (footer) повторить:

> "Помни: «завтра» = {tomorrow_date}, текущая дата для всех вычислений — {current_date}."

### 4. Расширение набора примеров

Добавить примеры для:
- Конец месяца: "завтра" при 31 марта → 1 апреля
- "Сегодня вечером" → текущая дата, 19:00
- Относительное время: "через 2 часа" при текущем 14:00 → 16:00
- Событие "в воскресенье" при текущей дате суббота → завтра (не через неделю)

Примеры будут автоматически корректны для любой текущей даты за счёт динамической генерации.

## Risks / Trade-offs

**[Риск] Усложнение кода промпта** → Генерация примеров добавляет ~40 строк кода вместо статической строки. Митигация: чёткая структура, каждый пример — декларативная структура.

**[Риск] Больший размер промпта → больше токенов** → Добавление 2-3 примеров увеличит промпт на ~100 токенов. Митигация: на фоне общего размера промпта (~800 токенов) это незначительно (+12%).

**[Trade-off] Тесты промпта становятся сложнее** → Сейчас примеры можно проверить визуально. С динамическими — нужны юнит-тесты. Это скорее плюс: тесты гарантируют корректность вычислений.
