## Context

Бот Shared Plan работает в режиме long-polling и доступен любому пользователю Telegram. Архитектура: middleware-цепочка (`DbSessionMiddleware` → `UserProfileMiddleware` → handlers). Пользователи идентифицируются по Telegram user ID. Авторизация существует только на уровне пространств (admin/member), но не на уровне бота.

Бот предназначен для семейного использования. Необходимо ограничить доступ на уровне бота: только владелец и приглашённые через пространство пользователи.

## Goals / Non-Goals

**Goals:**
- Блокировать взаимодействие с ботом для неавторизованных пользователей
- Определять владельца бота через конфигурацию
- Использовать существующий механизм space invite links как способ авторизации
- Минимальное влияние на существующий код и производительность

**Non-Goals:**
- Отдельные bot-level invite-ссылки (достаточно существующих space invites)
- Ручное управление whitelist-ом (approve/revoke команды)
- Временные инвайты с истечением срока
- Rate limiting или anti-abuse механизмы

## Decisions

### 1. Членство в пространстве = авторизация в боте

**Решение**: Не вводить новую колонку `is_approved` в таблице `users`. Авторизация определяется наличием хотя бы одной записи в `user_spaces` для данного `user_id`, либо совпадением с `OWNER_TELEGRAM_ID`.

**Альтернатива**: Добавить `is_approved: bool` в `users` — отвергнуто, т.к. создаёт необходимость синхронизации с `user_spaces` (при kick/delete). Текущие данные `user_spaces` уже содержат всю нужную информацию.

**Обоснование**: Простота, отсутствие рассинхронизации, нет миграции БД. Запрос `SELECT 1 FROM user_spaces WHERE user_id = :uid LIMIT 1` — O(1) по составному PK.

### 2. Middleware на уровне update

**Решение**: Новый `AccessControlMiddleware`, зарегистрированный на `dp.update.middleware()`.

**Порядок middleware:**
1. `DbSessionMiddleware` — создаёт async session
2. `AccessControlMiddleware` — проверяет доступ (нужна session)
3. `UserProfileMiddleware` — upsert профиля (только для авторизованных)

**Обоснование**: Middleware на уровне update перехватывает ВСЕ типы событий (message, callback_query, inline_query и т.д.). Размещение перед `UserProfileMiddleware` означает, что для заблокированных пользователей не будет создаваться/обновляться профиль.

### 3. Пропуск deep link join для неавторизованных

**Решение**: Middleware пропускает обработку, если update содержит message с текстом `/start join_*`. Это позволяет новому пользователю присоединиться к пространству по invite-ссылке.

**Логика проверки в middleware:**
1. Извлечь `user_id` из update
2. Если `user_id == OWNER_TELEGRAM_ID` → пропустить (allow)
3. Если update — это `/start join_*` → пропустить (allow join flow)
4. Если существует запись в `user_spaces` для `user_id` → пропустить (allow)
5. Иначе → отправить сообщение «Бот доступен только по приглашению» и прервать обработку

### 4. OWNER_TELEGRAM_ID как обязательный параметр

**Решение**: Новый обязательный env var `OWNER_TELEGRAM_ID` (int). Владелец всегда авторизован, даже если не состоит ни в одном пространстве.

**Альтернатива**: Определять владельца как первого зарегистрированного пользователя — отвергнуто: неявно, может привести к ошибке если бота запустил не тот человек первым.

## Risks / Trade-offs

- **[Дополнительный запрос к БД на каждый update]** → Запрос по PK-индексу `user_spaces`, минимальная нагрузка. Для владельца запрос не выполняется (проверка по ID в памяти).
- **[Потеря доступа при kick из последнего пространства]** → Ожидаемое поведение для семейного бота. Владелец может повторно пригласить пользователя.
- **[Невозможность /start без join-кода для новых пользователей]** → Для семейного бота это фича, а не баг. Новый пользователь получает invite-ссылку от владельца.
