## Context

Проект Shared Plan Bot начинается с чистого листа. Необходимо создать фундамент, на котором будут строиться все последующие фазы: пространства (Фаза 2), события (Фаза 3), AI-парсинг (Фаза 4), голос (Фаза 5), напоминания (Фаза 6).

Целевая архитектура определена в `docs/architecture.md`: однопроцессный async Python-бот, работающий в Docker Compose вместе с PostgreSQL 16. Модель данных описана в `docs/datamodel.md`: 5 таблиц с предопределёнными типами, индексами и каскадными правилами.

Фаза 1 создаёт весь скелет, включая таблицы, которые будут использоваться в последующих фазах. Это осознанное решение: одна миграция для всей схемы проще, чем инкрементальные миграции на каждую фазу.

## Goals / Non-Goals

**Goals:**

- Работающий `docker compose up` → PostgreSQL + бот с long-polling
- Все 5 таблиц БД созданы через Alembic миграцию
- `/start` создаёт пользователя (upsert), `/help` показывает команды
- Конфигурация валидируется при старте через pydantic-settings
- Middleware внедряет DB-сессию в каждый handler

**Non-Goals:**

- Бизнес-логика пространств, событий, напоминаний (Фазы 2-6)
- AI-парсинг и голосовые сообщения (Фазы 4-5)
- APScheduler задачи (Фаза 6; сам APScheduler ставится как зависимость, но не запускается)
- Тесты, CI/CD, линтинг
- Hot reload в Docker (volume mount для кода — достаточно ручного `docker compose restart`)

## Decisions

### 1. Все таблицы в начальной миграции

Создаём все 5 таблиц (`users`, `spaces`, `user_spaces`, `events`, `scheduled_reminders`) в одной Alembic-миграции, хотя таблицы событий и напоминаний не используются до Фаз 3 и 6.

**Альтернатива**: инкрементальные миграции в каждой фазе.
**Почему отвергнута**: модель данных стабильна и полностью определена заранее. Одна миграция проще в управлении. Пустые таблицы не мешают работе.

### 2. aiogram 3.x с MemoryStorage для FSM

FSM-состояния хранятся в памяти (стандартный `MemoryStorage`). Состояния не переживают перезапуск — это допустимо для прерванных многошаговых флоу.

**Альтернатива**: Redis для FSM-хранилища.
**Почему отвергнута**: избыточная зависимость для личного бота. Потеря FSM-состояния при перезапуске — незначительная проблема.

### 3. Middleware для DB-сессии

Каждый handler получает SQLAlchemy async-сессию через aiogram middleware. Сессия создаётся в начале обработки, коммитится при успехе, откатывается при исключении.

**Альтернатива**: контекстный менеджер сессии внутри каждого handler.
**Почему отвергнута**: дублирование boilerplate. Middleware обеспечивает единообразное управление транзакциями.

### 4. Upsert пользователя в /start

При каждом `/start` — `INSERT ... ON CONFLICT (id) DO UPDATE SET first_name, username`. Это обновляет отображаемое имя при каждом взаимодействии.

**Альтернатива**: только INSERT, без обновления.
**Почему отвергнута**: пользователи меняют имена и username в Telegram. Upsert дёшев и поддерживает актуальность данных.

### 5. Структура проекта

Плоская структура внутри `app/`: `bot/handlers/`, `bot/keyboards/`, `bot/callbacks/`, `bot/states/`, `bot/middlewares/`, `services/`, `db/`, `scheduler/`. Каждый модуль — один файл с узкой ответственностью.

**Альтернатива**: единый файл `bot.py` с постепенным разделением.
**Почему отвергнута**: structure.md из `docs/architecture.md` уже определяет модульную структуру. Лучше создать скелет сразу, чем рефакторить позже.

### 6. requirements.txt с зависимостями всех фаз

Включаются все зависимости, которые понадобятся во всех фазах (openai, aiohttp, apscheduler), чтобы не пересобирать Docker-образ на каждой фазе. Версии фиксируются точно где возможно; для пакетов с конфликтами между зависимостями (pydantic, aiohttp, openai) используются совместимые диапазоны — aiogram 3.15.0 ограничивает `pydantic<2.10` и `aiohttp<3.11`.

## Risks / Trade-offs

**[Пустые таблицы в БД]** → Таблицы `events` и `scheduled_reminders` будут пустыми до Фаз 3 и 6. Не влияет на работу, но может вызвать вопросы при инспекции БД. Митигация: документация в CLAUDE.md объясняет поэтапную разработку.

**[Все зависимости сразу в requirements.txt]** → Docker-образ содержит неиспользуемые пакеты (openai, aiohttp для Nexara). Увеличивает размер образа незначительно. Митигация: упрощает workflow — не нужно пересобирать образ при переходе к новым фазам.

**[FSM в памяти]** → Перезапуск бота сбрасывает все активные FSM-сессии. Митигация: FSM используется только для коротких флоу (создание пространства), пользователь просто повторит команду.
