## ADDED Requirements

### Requirement: Карточка управления событием

Владелец события ДОЛЖЕН иметь возможность открыть карточку управления своим событием из списка `/events`.

#### Scenario: Открытие карточки управления
- **WHEN** владелец нажимает кнопку «Управление» рядом со своим событием в списке `/events`
- **THEN** бот показывает карточку с информацией о событии (название, дата, время) и inline-кнопками «Изменить» и «Удалить»

#### Scenario: Кнопка управления видна только владельцу
- **WHEN** пользователь просматривает список `/events` и событие создано другим пользователем
- **THEN** кнопка «Управление» для этого события НЕ отображается

#### Scenario: Событие уже удалено
- **WHEN** пользователь нажимает кнопку «Управление» для события, которое уже было удалено
- **THEN** бот отвечает callback answer «Событие не найдено»

### Requirement: Удаление события владельцем

Владелец события ДОЛЖЕН иметь возможность удалить своё событие с подтверждением действия.

#### Scenario: Запрос подтверждения удаления
- **WHEN** владелец нажимает кнопку «Удалить» в карточке управления
- **THEN** бот показывает сообщение «Удалить событие «{title}»? Это действие нельзя отменить.» с кнопками «Да, удалить» и «Отмена»

#### Scenario: Подтверждение удаления
- **WHEN** владелец нажимает «Да, удалить»
- **THEN** событие удаляется из БД, связанные напоминания удаляются каскадно, участники пространства (кроме владельца) получают уведомление об удалении, бот показывает сообщение «Событие удалено»

#### Scenario: Отмена удаления
- **WHEN** владелец нажимает «Отмена» в диалоге подтверждения удаления
- **THEN** диалог закрывается, событие остаётся без изменений

#### Scenario: Удаление несуществующего события
- **WHEN** владелец подтверждает удаление события, которое уже было удалено
- **THEN** бот отвечает callback answer «Событие не найдено»

### Requirement: Редактирование названия события

Владелец события ДОЛЖЕН иметь возможность изменить название своего события.

#### Scenario: Выбор поля «Название» для редактирования
- **WHEN** владелец нажимает «Изменить» в карточке управления и выбирает «Название»
- **THEN** бот просит ввести новое название текстовым сообщением

#### Scenario: Успешное изменение названия
- **WHEN** владелец вводит новое название (1-500 символов)
- **THEN** бот показывает подтверждение с текущим и новым названием; после подтверждения название обновляется, участники получают уведомление об изменении

#### Scenario: Слишком длинное название
- **WHEN** владелец вводит название длиннее 500 символов
- **THEN** бот сообщает об ошибке и просит ввести название короче

### Requirement: Редактирование даты события

Владелец события ДОЛЖЕН иметь возможность изменить дату своего события.

#### Scenario: Выбор поля «Дата» для редактирования
- **WHEN** владелец нажимает «Изменить» в карточке управления и выбирает «Дата»
- **THEN** бот просит ввести новую дату текстовым сообщением (на естественном языке)

#### Scenario: Успешное изменение даты
- **WHEN** владелец вводит новую дату и она успешно распарсена через LLM
- **THEN** бот показывает подтверждение с текущей и новой датой; после подтверждения дата обновляется, неотправленные напоминания пересоздаются, участники получают уведомление

#### Scenario: Ошибка парсинга даты
- **WHEN** LLM не может распарсить введённую дату
- **THEN** бот показывает сообщение об ошибке парсинга и предлагает повторить ввод

### Requirement: Редактирование времени события

Владелец события ДОЛЖЕН иметь возможность изменить или убрать время своего события.

#### Scenario: Выбор поля «Время» для редактирования
- **WHEN** владелец нажимает «Изменить» в карточке управления и выбирает «Время»
- **THEN** бот просит ввести новое время текстовым сообщением или нажать кнопку «Убрать время» (для события на весь день)

#### Scenario: Успешное изменение времени
- **WHEN** владелец вводит новое время и оно успешно распарсено через LLM
- **THEN** бот показывает подтверждение с текущим и новым временем; после подтверждения время обновляется, неотправленные напоминания пересоздаются, участники получают уведомление

#### Scenario: Удаление времени (событие на весь день)
- **WHEN** владелец нажимает «Убрать время»
- **THEN** время события устанавливается в NULL, неотправленные напоминания пересоздаются (только «1d» для событий без времени), участники получают уведомление

#### Scenario: Ошибка парсинга времени
- **WHEN** LLM не может распарсить введённое время
- **THEN** бот показывает сообщение об ошибке парсинга и предлагает повторить ввод

### Requirement: Уведомления участников при изменении/удалении

Все участники пространства (кроме владельца события) ДОЛЖНЫ получать уведомление при редактировании или удалении события.

#### Scenario: Уведомление при редактировании
- **WHEN** владелец изменяет любое поле события
- **THEN** всем участникам пространства (кроме владельца) отправляется сообщение с указанием что изменилось, новым значением и названием пространства

#### Scenario: Уведомление при удалении
- **WHEN** владелец удаляет событие
- **THEN** всем участникам пространства (кроме владельца) отправляется сообщение с названием удалённого события и названием пространства

#### Scenario: Ошибка отправки уведомления
- **WHEN** уведомление не удаётся отправить одному из участников (бот заблокирован и т.д.)
- **THEN** ошибка логируется, остальные уведомления продолжают отправляться

### Requirement: Пересоздание напоминаний при изменении даты/времени

При изменении даты или времени события ВСЕ неотправленные напоминания ДОЛЖНЫ быть пересозданы с учётом новых значений.

#### Scenario: Пересоздание напоминаний после изменения даты
- **WHEN** владелец изменяет дату события
- **THEN** все напоминания с `sent=FALSE` удаляются, новые напоминания создаются по настройкам каждого участника пространства на основе новой даты

#### Scenario: Пересоздание напоминаний после изменения времени
- **WHEN** владелец изменяет время события
- **THEN** все напоминания с `sent=FALSE` удаляются, новые напоминания создаются по настройкам каждого участника пространства на основе нового времени

#### Scenario: Напоминания не пересоздаются при изменении названия
- **WHEN** владелец изменяет только название события
- **THEN** напоминания остаются без изменений (название подтягивается из события при отправке)

### Requirement: Только будущие события доступны для управления

Кнопки управления ДОЛЖНЫ отображаться только для будущих событий.

#### Scenario: Будущее событие
- **WHEN** событие с `event_date >= текущая дата`
- **THEN** кнопка «Управление» отображается для владельца

#### Scenario: Прошедшее событие
- **WHEN** событие с `event_date < текущая дата`
- **THEN** кнопка «Управление» НЕ отображается
