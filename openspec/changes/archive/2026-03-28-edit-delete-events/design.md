## Context

Сейчас события в SharedPlan — неизменяемые: после публикации ни название, ни дату/время нельзя поправить, а отменённое событие нельзя удалить. Владелец (created_by) никак не выделен в UX после создания.

Существующая схема `events` уже содержит `created_by` (BIGINT FK → users), а `scheduled_reminders` настроены с `ON DELETE CASCADE` по `event_id`, что упрощает удаление.

## Goals / Non-Goals

**Goals:**
- Владелец может удалить своё событие (с подтверждением)
- Владелец может отредактировать название, дату, время своего события
- При изменении даты/времени напоминания пересоздаются для всех участников пространства
- Участники получают уведомления об изменениях и удалении
- Кнопки управления доступны из списка `/events`

**Non-Goals:**
- Редактирование/удаление событий другими участниками (только владелец)
- Передача владения событием другому пользователю
- История изменений / аудит-лог
- Массовое удаление или редактирование
- Редактирование прошедших событий (нет смысла)

## Decisions

### 1. Точка входа — inline-кнопки в списке `/events`

**Решение**: после каждого события в списке, если текущий пользователь — его владелец, добавлять inline-кнопку «Управление». По нажатию — показывать карточку события с кнопками «Изменить» и «Удалить».

**Альтернатива**: отдельная команда `/manage` с выбором события из списка. Отклонена: добавляет лишний шаг навигации, пользователь уже видит свои события в `/events`.

**Callback data**: `event_manage:{event_id}` → показать карточку с кнопками.

### 2. Удаление — два шага с подтверждением

**Решение**: кнопка «Удалить» → подтверждение «Удалить событие {title}? Это действие нельзя отменить» с кнопками «Да, удалить» / «Отмена».

**Callback data**: `event_delete:{event_id}` → показать подтверждение, `event_delete_yes:{event_id}` → удалить, `event_delete_no:{event_id}` → отмена.

При удалении: `scheduled_reminders` удаляются каскадно через FK. Уведомление отправляется всем участникам пространства (кроме владельца).

### 3. Редактирование — выбор поля → ввод нового значения через FSM

**Решение**: кнопка «Изменить» → inline-клавиатура с выбором поля (Название / Дата / Время) → FSM-состояние ожидания нового значения → подтверждение → сохранение.

**FSM-состояния**:
- `EditEvent.waiting_for_field` — выбор поля для изменения
- `EditEvent.waiting_for_value` — ожидание нового значения
- `EditEvent.waiting_for_confirm` — подтверждение изменения

**Callback data**: `event_edit_field:{event_id}:{field}` где field = `title`, `date`, `time`.

При изменении даты/времени: удалить все неотправленные напоминания (`sent=FALSE`) → пересоздать напоминания с новыми значениями через `reminder_service.create_reminders_for_event`.

При изменении названия: напоминания не пересоздаются (они ссылаются на event, название подтягивается при отправке).

### 4. Парсинг нового значения даты/времени — через LLM

**Решение**: при редактировании даты или времени новое значение парсится через тот же `llm_parser`, но с упрощённым промптом, извлекающим только дату или только время.

**Альтернатива**: ожидать строгий формат (ДД.ММ.ГГГГ, ЧЧ:ММ). Отклонена: противоречит UX проекта, где пользователь пишет естественным языком.

### 5. Проверка владения — на уровне сервиса

**Решение**: `event_service.get_event_for_owner(session, event_id, user_id)` возвращает событие, только если `created_by == user_id`. Хендлеры не делают проверку сами — это ответственность сервиса.

### 6. Уведомления участников

**Решение**: при изменении/удалении события — уведомить всех участников пространства, кроме владельца, о том, что произошло. Формат уведомления аналогичен существующему (`format_notification`), но с другим текстом: «Событие изменено» / «Событие удалено».

## Risks / Trade-offs

- **Race condition при одновременном удалении и отправке напоминания** → Минимальный риск: планировщик помечает `sent=TRUE` перед отправкой, а CASCADE-удаление просто не найдёт уже отправленные записи. Неотправленные удалятся корректно.
- **Пользователь нажимает старую inline-кнопку после удаления события** → Обработчик проверяет существование события и возвращает callback answer «Событие не найдено».
- **LLM-парсинг одного поля менее надёжен, чем полного текста** → Используется специальный упрощённый промпт; при ошибке парсинга — стандартное сообщение об ошибке с предложением повторить.
