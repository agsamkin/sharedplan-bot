# Shared Plan Bot — Фазы разработки

**Документ разработки для OpenSpec-подхода**

> Каждая фаза — изолированный, запускаемый кусок работы. Фазы идут от ядра к периферии. Каждая следующая фаза опирается на предыдущие.

---

## Инфраструктурный стек

| Компонент | Решение |
|---|---|
| Dev-окружение | docker-compose (Python-бот + PostgreSQL 16) |
| Telegram-фреймворк | aiogram 3.x (async, long-polling) |
| Язык | Python 3.12+ |
| БД | PostgreSQL 16 в Docker |
| ORM | SQLAlchemy 2.0 (async) + asyncpg, миграции Alembic |
| LLM (парсинг текста) | OpenRouter API (OpenAI-совместимый, конфигурируемая модель) |
| Speech-to-Text | Nexara API (`api.nexara.ru/api/v1/audio/transcriptions`) |
| Планировщик | APScheduler 3.x (AsyncIOScheduler) |
| HTTP-клиент | aiohttp (async), openai SDK (для OpenRouter) |
| Конфигурация | pydantic-settings (.env) |

---

## Фаза 1 — Скелет: бот + БД + Docker + конфигурация

**Что делаем:**
- Структура проекта на Python 3.12+ (модульная: `bot/handlers/`, `bot/keyboards/`, `bot/callbacks/`, `bot/states/`, `bot/middlewares/`, `services/`, `db/`, `scheduler/`)
- aiogram 3.x приложение с базовой конфигурацией: Bot, Dispatcher, Router
- docker-compose: Python-бот (hot reload через volume mount) + PostgreSQL 16 (persistent volume `postgres_data`)
- Dockerfile: `python:3.12-slim`, установка зависимостей через `requirements.txt`
- Alembic для миграций (async-конфигурация с asyncpg)
- Начальная миграция — все таблицы в одной транзакции:
  - `users` (id BIGINT PK — Telegram ID, username VARCHAR NULL, first_name VARCHAR, reminder_settings JSONB DEFAULT, created_at TIMESTAMPTZ)
  - `spaces` (id UUID PK, name VARCHAR, invite_code VARCHAR UNIQUE, created_by BIGINT FK → users, created_at TIMESTAMPTZ)
  - `user_spaces` (user_id BIGINT + space_id UUID — составной PK, role VARCHAR CHECK IN ('admin','member'), joined_at TIMESTAMPTZ) + ON DELETE CASCADE на обоих FK
  - `events` (id UUID PK, space_id UUID FK → spaces ON DELETE CASCADE, title VARCHAR, event_date DATE, event_time TIME NULL, created_by BIGINT FK → users, raw_input TEXT NULL, created_at TIMESTAMPTZ) + индекс `idx_events_space_date` на `(space_id, event_date)`
  - `scheduled_reminders` (id UUID PK, event_id UUID FK → events ON DELETE CASCADE, user_id BIGINT FK → users ON DELETE CASCADE, remind_at TIMESTAMPTZ, reminder_type VARCHAR, sent BOOLEAN DEFAULT FALSE) + индекс `idx_reminders_remind_at_sent` на `(remind_at, sent)`
- SQLAlchemy ORM-модели в `db/models.py`, async engine + session factory в `db/engine.py`
- Конфигурация через environment variables (`.env` + pydantic-settings): `TELEGRAM_BOT_TOKEN`, `DATABASE_URL`, `OPENROUTER_API_KEY`, `OPENROUTER_MODEL`, `NEXARA_API_KEY`, `TIMEZONE`, `REMINDER_CHECK_INTERVAL_SECONDS`
- Обработчик `/start` — приветственное сообщение, создание пользователя в БД (upsert по Telegram ID)
- Обработчик `/help` — список команд
- Middleware `db_session.py` — внедрение async-сессии SQLAlchemy в каждый handler
- `.env.example` со всеми переменными и комментариями
- `requirements.txt` с зафиксированными версиями

**Чего НЕ делаем:**
- Никакого AI-парсинга (OpenRouter — это Фаза 4)
- Никаких голосовых сообщений (Nexara — это Фаза 5)
- Никакой бизнес-логики (CRUD пространств — это Фаза 2, события — Фаза 3)
- Никаких напоминаний (APScheduler — это Фаза 6)

**Результат:** `docker compose up` → PostgreSQL стартует, Alembic создаёт все таблицы, бот запускается в long-polling режиме, `/start` отвечает приветствием и создаёт пользователя в БД, `/help` показывает список команд.

## Фаза 2 — Пространства: создание, приглашение, управление

**Что делаем:**
- FSM-состояние `CreateSpace.waiting_for_name` в `bot/states/create_space.py`
- Обработчик `/newspace`:
  - Бот спрашивает название пространства
  - Пользователь вводит название (1–255 символов, валидация)
  - Генерация `invite_code` через `secrets.token_urlsafe(6)`
  - Создание записей в `spaces` и `user_spaces` (role='admin') в одной транзакции
  - Ответ с подтверждением и invite-ссылкой: `https://t.me/<bot_username>?start=join_<invite_code>`
- Deep link обработка в `/start`:
  - Парсинг payload `join_<invite_code>` из параметра `/start`
  - Поиск пространства по `invite_code`
  - Добавление пользователя с role='member' (если ещё не участник)
  - Уведомление всех текущих участников о новом члене
  - Обработка ошибок: невалидный код, уже участник
- Обработчик `/spaces` — список пространств пользователя (название, роль, количество участников)
- Обработчик `/space_info` — информация о пространстве: список участников, invite-ссылка (с выбором пространства, если их несколько)
- Обработчик `/kick @username` — удаление участника (проверка роли admin)
- Обработчик `/delete_space` — удаление пространства с inline-подтверждением (проверка роли admin, каскадное удаление)
- Сервисный слой `services/space_service.py`: вся бизнес-логика пространств
- Inline-клавиатуры: `keyboards/space_select.py` (выбор пространства)
- Callback-обработчики: `callbacks/space_select.py`
- Фильтр `filters/not_command.py` — определяет, что сообщение не является командой (понадобится в Фазе 3)

**Чего НЕ делаем:**
- Никаких событий (Фаза 3)
- Никакого AI-парсинга
- Никаких напоминаний

**Результат:** полный жизненный цикл пространств: создание → invite-ссылка → присоединение по ссылке → просмотр участников → удаление участника → удаление пространства. Deep link работает через Telegram.

## Фаза 3 — События: создание из текста + подтверждение + уведомления

**Что делаем:**
- Обработчик текстовых сообщений (не команд) в `bot/handlers/event.py`:
  - Фильтр `not_command` перехватывает все обычные текстовые сообщения
  - Если пользователь в одном пространстве — используем его. Если в нескольких — inline-клавиатура выбора пространства
  - Заглушка парсинга: на этой фазе пользователь вводит событие в структурированном формате (название + дата + время через запятую), AI-парсинг — Фаза 4
- Карточка подтверждения с inline-кнопками `[✅ Да]` `[✏️ Изменить]` `[❌ Отмена]`:
  - `event_confirm:{uuid}` — сохранение события
  - `event_edit:{uuid}` — переход к редактированию (простой: бот просит отправить исправленный вариант)
  - `event_cancel:{uuid}` — отмена
- При подтверждении:
  - Событие сохраняется в `events`
  - Уведомления отправляются всем участникам пространства (кроме создателя)
  - Каждая отправка обёрнута в try/except — ошибка логируется, не прерывает остальных
- Обработчик `/events` — ближайшие 10 будущих событий пространства:
  - Сортировка по `event_date`, `event_time`
  - Относительные даты для ближайших событий (сегодня, завтра)
  - События без времени — без компонента «⏰»
- Сервисный слой `services/event_service.py`: создание события, получение списка, уведомления
- Inline-клавиатуры: `keyboards/confirm.py`
- Callback-обработчики: `callbacks/event_confirm.py`

**Чего НЕ делаем:**
- Никакого AI-парсинга естественного языка (заглушка с ручным форматом — AI в Фазе 4)
- Никаких голосовых сообщений (Фаза 5)
- Никаких напоминаний (Фаза 6) — события создаются, но `scheduled_reminders` пока не заполняется

**Результат:** события создаются через структурированный ввод, подтверждаются через inline-кнопки, сохраняются в БД, участники получают уведомления, `/events` показывает расписание. Весь флоу работает end-to-end без AI.

## Фаза 4 — AI-парсинг: OpenRouter + извлечение событий из текста

**Что делаем:**
- Сервисный слой `services/llm_parser.py`:
  - Клиент через `openai` SDK с `base_url="https://openrouter.ai/api/v1"`
  - Метод `parse_event(user_text, current_date, weekday)` → `{title, date, time}`
  - Системный промпт с текущей датой, днём недели, часовым поясом `Europe/Moscow`
  - `temperature: 0.1`, `response_format: {"type": "json_object"}`
  - Валидация ответа через Pydantic-схему `ParsedEvent(title: str, date: date, time: Optional[time])`
  - Retry: 3 попытки с экспоненциальным backoff при HTTP 429/5xx
  - Один повтор при невалидном JSON от LLM (с усиленной инструкцией)
  - Таймаут: 15 секунд на запрос
  - Логирование: модель, input_tokens, output_tokens, duration_ms
- Замена заглушки парсинга в `bot/handlers/event.py`:
  - Текстовое сообщение пользователя → `llm_parser.parse_event()` → карточка подтверждения
  - Обработка ошибок:
    - Невалидный JSON: «Не удалось распознать событие. Попробуй переформулировать.»
    - Таймаут/сеть: «Ошибка соединения, попробуй позже.»
    - Rate limit: retry автоматически, при исчерпании попыток — «Сервис временно недоступен.»
- Промпт в отдельном модуле (`prompts/event_parser.py`):
  - System prompt + шаблон user prompt + описание ожидаемого JSON
  - Примеры разрешения относительных дат (завтра, в понедельник, через 2 дня)
  - Инструкция: если время не указано — `time: null`

**Чего НЕ делаем:**
- Никаких голосовых сообщений (Фаза 5)
- Никакого тюнинга промптов (Фаза 8)
- Никаких напоминаний

**Результат:** пользователь пишет «Ужин с родителями завтра в 19:00» → LLM извлекает структурированные данные → бот показывает подтверждение → публикация. Естественный язык работает для русского и английского.

## Фаза 5 — Голосовые сообщения: Nexara STT + интеграция с event flow

**Что делаем:**
- Сервисный слой `services/speech_to_text.py`:
  - Метод `transcribe(audio_bytes, mime_type)` → `str`
  - `aiohttp` POST на `https://api.nexara.ru/api/v1/audio/transcriptions`
  - Multipart upload: `file` (audio bytes, MIME `audio/ogg`), `response_format: "json"`
  - Таймаут: 30 секунд (зависит от длины аудио)
  - Retry: 2 попытки с backoff при 5xx
  - Логирование: duration_ms, длина аудио, длина транскрипции
- Обработчик голосовых сообщений `bot/handlers/voice.py`:
  - Перехват `Message.voice`
  - Скачивание аудио через Telegram Bot API (`bot.download(voice.file_id)`) в memory buffer (не на диск)
  - Отправка в Nexara → получение транскрипции
  - Передача транскрипции в `llm_parser.parse_event()` (тот же флоу, что и для текста)
  - Карточка подтверждения с транскрипцией:
    ```
    🎤 Распознано: «Ужин с родителями завтра в семь вечера»

    📌 Событие:
    📝 Ужин с родителями
    📅 27 марта 2026
    ⏰ 19:00

    Опубликовать?
    ```
  - Обработка ошибок Nexara: «Не удалось распознать голос. Попробуй ещё раз или напиши текстом.»

**Чего НЕ делаем:**
- Никаких напоминаний (Фаза 6)
- Никакого тюнинга распознавания
- Никакой поддержки видеосообщений (только voice)

**Результат:** пользователь отправляет голосовое → бот транскрибирует через Nexara → парсит через OpenRouter → показывает подтверждение с транскрипцией. Полный voice-to-event пайплайн.

## Фаза 6 — Напоминания: планировщик + персональные настройки

**Что делаем:**
- APScheduler `AsyncIOScheduler` запускается в `main.py` вместе с ботом
- Интервальная задача `process_due_reminders` в `scheduler/jobs.py`:
  - Интервал: каждые N секунд (из `REMINDER_CHECK_INTERVAL_SECONDS`, по умолчанию 30)
  - Запрос: `SELECT * FROM scheduled_reminders WHERE sent = FALSE AND remind_at <= NOW() ORDER BY remind_at LIMIT 50`
  - Для каждого напоминания: загрузить событие → отправить Telegram-сообщение → `SET sent = TRUE`
  - Ошибки отправки: пометить sent=TRUE, залогировать (не повторять бесконечно)
- Создание напоминаний при публикации события (дополнение `services/event_service.py`):
  - При подтверждении события — итерация по всем участникам пространства
  - Для каждого участника: прочитать `reminder_settings`, для каждого включённого интервала — вычислить `remind_at`
  - Если `remind_at` уже в прошлом — не создавать
  - Для событий без времени — только `1d` (на 09:00 накануне)
  - Все записи `scheduled_reminders` создаются в той же транзакции, что и событие
- Сервисный слой `services/reminder_service.py`:
  - Расчёт `remind_at` из event_date + event_time + интервал
  - Запрос и отправка наступивших напоминаний
  - Форматирование сообщения напоминания:
    ```
    🔔 Напоминание!

    📝 Ужин с родителями
    ⏰ Через 15 минут (19:00)
    📍 Пространство: Семья
    ```
- Обработчик `/reminders` — настройки напоминаний:
  - Показ текущих настроек с inline toggle-кнопками (☑ / ☐)
  - Callback `reminder_toggle:{key}` — переключение интервала в `users.reminder_settings`
  - Обновление сообщения после каждого нажатия (edit message + update keyboard)
- Inline-клавиатура: `keyboards/reminder_settings.py`
- Callback-обработчик: `callbacks/reminder_toggle.py`

**Чего НЕ делаем:**
- Никакого тюнинга интервалов проверки
- Никаких per-event настроек напоминаний
- Никакого retry для неотправленных (sent=TRUE при ошибке — финальное решение)

**Результат:** при публикации события автоматически создаются напоминания для всех участников. Планировщик каждые 30 секунд отправляет наступившие. Каждый участник настраивает свои интервалы через `/reminders`. Полный цикл: событие → напоминания → доставка.

## Фаза 7 — Надёжность: error handling + edge cases + логирование

**Что делаем:**
- Graceful degradation AI-пайплайна:
  - Таймаут на LLM-вызов: 15 секунд, на Nexara: 30 секунд
  - При ошибке парсинга JSON от LLM — одна повторная попытка с усиленной инструкцией
  - При полном отказе — понятное сообщение пользователю, не traceback
- Обработка edge cases:
  - Пользователь не состоит ни в одном пространстве → «Сначала создай пространство через /newspace»
  - Пользователь заблокировал бота → try/except при отправке, логирование, продолжение рассылки остальным
  - Invite-код не найден → «Ссылка недействительна или пространство удалено»
  - Пустое голосовое сообщение / тишина → обработка пустого текста от Nexara
  - Событие в прошлом → предупреждение: «Дата уже прошла. Всё равно создать?»
- Валидация входных данных:
  - Название пространства: 1–255 символов
  - Текст события: 1–1000 символов
  - Ответ LLM: Pydantic-схема `ParsedEvent`
  - Дата события: валидная дата, не более 2 лет в будущее
- Структурированное логирование:
  - Все LLM-вызовы: model, tokens_in, tokens_out, duration_ms
  - Все Nexara-вызовы: audio_duration, transcription_length, duration_ms
  - Все ошибки отправки Telegram: user_id, error_type
  - Все команды: user_id, command, space_id (где применимо)
- Обновление first_name пользователя при каждом взаимодействии (отражение актуального имени из Telegram)
- Тест конфигурации при старте: проверка всех обязательных переменных, проверка связности с PostgreSQL, проверка валидности Telegram-токена (getMe)

**Чего НЕ делаем:**
- Никакого rate limiting (личный бот)
- Никакого Sentry или внешнего мониторинга
- Никакого нагрузочного тестирования

**Результат:** бот устойчив к таймаутам LLM, невалидным JSON-ответам, заблокированным пользователям, некорректному вводу. Все ошибки логируются, пользователь всегда получает понятное сообщение.

## Фаза 8 — Тюнинг: промпты + UX + финальная полировка

**Что делаем:**
- Тестирование AI-парсинга на 20+ примерах:
  - Простые: «Ужин завтра в 19:00»
  - Относительные: «Тренировка в понедельник в 8 утра», «Встреча через 3 дня в 14:30»
  - Без времени: «День рождения Ани 5 апреля»
  - Сложные: «Стоматолог послезавтра утром», «Новый год», «Созвон с Петей на следующей неделе в среду после обеда»
  - Голосовые: те же примеры через голос (проверка цепочки Nexara → OpenRouter)
- Итеративное улучшение промпта `prompts/event_parser.py`:
  - Few-shot примеры для edge cases
  - Уточнение инструкций по разрешению неоднозначностей (например, «утром» → 09:00, «после обеда» → 14:00)
  - Настройка temperature (0.05 vs 0.1 vs 0.2) для баланса точности и гибкости
- Тюнинг UX сообщений:
  - Формулировки подтверждений, уведомлений, напоминаний
  - Форматирование дат (сегодня/завтра vs числовое)
  - Текст ошибок (дружелюбный, без технических деталей)
- Тестирование полного цикла:
  - Создание пространства → приглашение → создание события (текст) → уведомление → напоминание
  - Создание пространства → приглашение → создание события (голос) → уведомление → напоминание
  - Множественные пространства → выбор → публикация
  - Удаление пространства → каскадное удаление событий и напоминаний
- Сравнение моделей OpenRouter: gpt-4o-mini vs claude-haiku vs другие — по качеству парсинга и стоимости
- Проверка: бот корректно работает после docker compose restart (миграции, планировщик, polling)

**Чего НЕ делаем:**
- Никакого автоматического тестирования промптов
- Никаких юнит/интеграционных тестов (личный проект)
- Никакой оптимизации производительности

**Результат:** стабильный парсинг событий из естественного языка и голоса. Промпты отточены, формулировки сообщений проверены, полный цикл работает без ошибок. Бот готов к ежедневному использованию.

## Карта зависимостей между фазами

```
Фаза 1 (скелет: бот + БД + Docker)
  └── Фаза 2 (пространства: создание + invite + управление)
        └── Фаза 3 (события: создание + подтверждение + уведомления)
              ├── Фаза 4 (AI-парсинг: OpenRouter)  ← заменяет заглушку из Фазы 3
              │     └── Фаза 5 (голос: Nexara STT)  ← зависит от Фазы 4
              └── Фаза 6 (напоминания: APScheduler + настройки)  ← зависит от Фазы 3
                    └── Фаза 7 (надёжность: error handling + edge cases)
                          └── Фаза 8 (тюнинг: промпты + UX + полировка)
```

Фазы 4–5 и 6 можно делать параллельно — они независимы друг от друга (4–5 добавляют AI-парсинг, 6 добавляет напоминания; обе зависят от Фазы 3).
