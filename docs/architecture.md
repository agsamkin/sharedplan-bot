# Shared Plan Bot — Архитектура

## Область действия

Этот документ описывает **технические ограничения и архитектурные решения** Shared Plan Bot. Он определяет, что система должна обрабатывать, какие гарантии предоставлять, от каких интеграций зависеть и как компоненты связаны между собой.

---

## 1. Ограничения развёртывания

### Docker Compose стек

Система **должна** разворачиваться одной командой `docker compose up`, запускающей два сервиса:

| Сервис | Образ | Роль |
|---|---|---|
| `bot` | Собственный Dockerfile (Python 3.12-slim) | Приложение: Telegram-бот, планировщик, Mini App web-сервер, вся бизнес-логика |
| `db` | `postgres:16-alpine` | Хранение данных |

Других сервисов не требуется. Никаких внешних брокеров сообщений, Redis или отдельных воркер-процессов.

### Порядок запуска

Последовательность запуска **должна** быть следующей:

1. PostgreSQL запускается и становится доступен (health check через `pg_isready`).
2. Сервис бота запускается (`depends_on` с условием health check).
3. Alembic применяет все ожидающие миграции к базе данных.
4. APScheduler запускает задачу проверки напоминаний.
5. aiohttp web-сервер Mini App запускается на `MINI_APP_PORT`.
6. aiogram начинает long-polling к Telegram Bot API.

Если любой шаг завершается с ошибкой, процесс должен завершиться с ненулевым кодом возврата и понятным сообщением об ошибке.

### Конфигурация

Вся конфигурация передаётся через переменные окружения, загружаемые из файла `.env`. Модуль `config.py` на базе `pydantic-settings` валидирует и типизирует все значения при старте. Отсутствие обязательных переменных вызывает немедленную ошибку запуска с описательным сообщением — а не runtime-падение позже.

Обязательные переменные:

| Переменная | Назначение |
|---|---|
| `TELEGRAM_BOT_TOKEN` | Аутентификация в Telegram Bot API |
| `DATABASE_URL` | Строка подключения к PostgreSQL (async: `postgresql+asyncpg://...`) |
| `OPENROUTER_API_KEY` | Аутентификация в LLM API |
| `OPENROUTER_MODEL` | Идентификатор модели (например, `openai/gpt-4o-mini`) |
| `NEXARA_API_KEY` | Аутентификация в Speech-to-Text API |
| `OWNER_TELEGRAM_ID` | Telegram ID владельца бота (whitelist для контроля доступа) |

Опциональные переменные:

| Переменная | По умолчанию | Назначение |
|---|---|---|
| `TIMEZONE` | `Europe/Moscow` | Часовой пояс для всех операций с датой/временем |
| `REMINDER_CHECK_INTERVAL_SECONDS` | `30` | Частота проверки напоминаний планировщиком |
| `MINI_APP_URL` | — | URL для кнопок WebApp в Telegram |
| `MINI_APP_PORT` | `8080` | Порт aiohttp-сервера Mini App |

### Персистентность

Данные PostgreSQL **должны** переживать перезапуск контейнеров. Именованный Docker volume (`postgres_data`) монтируется в директорию данных PostgreSQL. Никакое состояние приложения не хранится на файловой системе контейнера бота — всё находится в базе данных.

## 2. Ограничения среды выполнения

### Асинхронная модель конкурентности

Все I/O-операции **должны** быть неблокирующими. Система работает в рамках одного asyncio event loop, который обрабатывает:

- Входящие обновления Telegram (через long-polling диспатчер aiogram).
- Исходящие вызовы Telegram API (отправка сообщений, скачивание файлов).
- HTTP-запросы к API OpenRouter и Nexara.
- Запросы и записи в базу данных (через SQLAlchemy async + asyncpg).
- Периодические проверки напоминаний APScheduler.
- HTTP-запросы к Mini App web-серверу (aiohttp).

Блокировка любой из этих операций остановит весь бот. Все внешние вызовы используют `aiohttp` или `asyncpg` — никаких синхронных HTTP-клиентов или драйверов БД.

### Однопроцессная архитектура

Бот работает как **один Python-процесс**. Нет отдельного воркера, Celery или фонового процесса. Диспатчер aiogram, APScheduler, aiohttp web-сервер Mini App и все обработчики сосуществуют в одном asyncio event loop. Это упрощает развёртывание и исключает межпроцессную координацию, но означает:

- Общая пропускная способность ограничена ёмкостью одного event loop (достаточно для личного бота).
- CPU-тяжёлая операция заблокирует всё (таких операций в системе нет — вся работа I/O-bound).

### Изоляция ошибок

Сбой одной операции **не должен** каскадировать на другие:

- Неудачная отправка уведомления одному пользователю не должна препятствовать уведомлениям другим. Каждая отправка обёрнута в try/except с логированием ошибки.
- Неудачная транскрипция Nexara должна привести к понятному сообщению пользователю, а не к падению.
- Неудачный парсинг LLM должен привести к сообщению «не удалось распознать», а не к необработанному исключению.
- Неудачная отправка напоминания должна пометить напоминание как отправленное (чтобы предотвратить бесконечные повторы) и залогировать ошибку.

## 3. Контракты интеграций

### Telegram Bot API (через aiogram 3.x)

Бот работает в режиме **long-polling** (не webhooks). Это исключает необходимость в публичном URL, TLS-сертификатах или HTTP-сервере — подходит для личного развёртывания.

**Правила обработки сообщений**:

- Команды (`/start`, `/help` и т.д.) маршрутизируются к выделенным обработчикам через роутер aiogram.
- Текстовые сообщения, не являющиеся командами, перехватываются обработчиком, инициирующим флоу создания события.
- Голосовые сообщения запускают пайплайн speech-to-text → создание события.
- Callback query от inline-кнопок маршрутизируются по префиксу callback data.

**Deep links**: Команда `/start` с payload, соответствующим `join_<invite_code>`, запускает флоу присоединения к пространству вместо приветственного сообщения.

**FSM (конечный автомат)**: Многошаговые взаимодействия (например, создание пространства — ожидание названия) используют встроенный FSM aiogram с хранением состояний в памяти. Состояние FSM не переживает перезапуск бота, что допустимо — прерванные многошаговые флоу просто истекают.

**Middleware-стек**: Три middleware обрабатывают каждое обновление в следующем порядке:

1. **`DbSessionMiddleware`** — внедряет async-сессию SQLAlchemy. Сессия коммитится при успехе и откатывается при исключении.
2. **`AccessControlMiddleware`** — контроль доступа в режиме invite-only. Пропускает только владельца бота (`OWNER_TELEGRAM_ID`) и участников хотя бы одного пространства. Исключение: `/start join_*` (присоединение по invite-ссылке) доступно всем. При ошибке проверки — fail-closed (доступ блокируется).
3. **`UserProfileMiddleware`** — upsert пользователя в БД при каждом взаимодействии (обновляет `first_name`, `username`). Работает после проверки доступа.

### OpenRouter API (LLM-парсинг текста)

**Endpoint**: `POST https://openrouter.ai/api/v1/chat/completions`

**Аутентификация**: `Authorization: Bearer {OPENROUTER_API_KEY}`

**Контракт**: Система отправляет структурированный промпт с сообщением пользователя и контекстом текущей даты/дня недели. LLM должна вернуть JSON-объект с тремя полями:

```json
{"title": "строка", "date": "YYYY-MM-DD", "time": "HH:MM или null"}
```

**Требования к промпту**:

- Системный промпт должен явно указывать текущую дату и день недели для разрешения относительных ссылок («завтра», «в понедельник»).
- Системный промпт должен указывать часовой пояс (`Europe/Moscow`).
- `temperature` должен быть `0.1` для минимизации «творческой» интерпретации.
- `response_format: {"type": "json_object"}` должен использоваться для гарантии JSON-вывода.
- Ответ должен валидироваться Pydantic-схемой. Невалидный JSON или отсутствующие поля приводят к сообщению пользователю: «Не удалось распознать событие. Попробуй переформулировать.»

**Реализация клиента**: Используется Python SDK `openai` с кастомным `base_url`, указывающим на `https://openrouter.ai/api/v1`. Это работает, потому что OpenRouter предоставляет OpenAI-совместимый API.

**Обработка ошибок**:

| HTTP-статус | Поведение |
|---|---|
| 429 (rate limit) | Повтор с экспоненциальным backoff, максимум 3 попытки |
| 5xx | Сообщение пользователю: «Сервис временно недоступен» |
| 200, но невалидный JSON | Сообщение пользователю: «Не удалось распознать событие» |
| Ошибка сети | Сообщение пользователю: «Ошибка соединения, попробуй позже» |

### Nexara API (Speech-to-Text)

**Endpoint**: `POST https://api.nexara.ru/api/v1/audio/transcriptions`

**Аутентификация**: `Authorization: Bearer {NEXARA_API_KEY}`

**Контракт**: Система отправляет аудиофайл как `multipart/form-data` с `response_format=json`. Nexara возвращает:

```json
{"text": "распознанный текст"}
```

**Ограничения ввода**:

- Голосовые сообщения Telegram приходят в формате OGG/Opus — Nexara поддерживает этот формат нативно.
- Максимальный размер файла, принимаемый Nexara: 1 ГБ (голосовые сообщения Telegram значительно меньше).

**Реализация клиента**: Используется `aiohttp` для асинхронной multipart-загрузки. Байты аудио читаются из файлового API Telegram и пересылаются напрямую в Nexara без записи на диск.

**Обработка ошибок**: Любой ответ, отличный от 200, приводит к сообщению пользователю: «Не удалось распознать голос. Попробуй ещё раз или напиши текстом.»

### PostgreSQL (через SQLAlchemy 2.0 Async + asyncpg)

**Подключение**: Async engine через `create_async_engine` с драйвером `postgresql+asyncpg`.

**Управление сессиями**: `async_sessionmaker` создаёт сессии, внедряемые в обработчики через middleware. Каждый обработчик работает в рамках одной сессии/транзакции.

**Миграции**: Alembic с поддержкой async. Миграции применяются при запуске до начала обработки обновлений. Целевая версия всегда `head`.

**Пул соединений**: Стандартные настройки пула asyncpg (min=1, max=10) достаточны для однопроцессного личного бота.

## 4. Архитектура фоновых задач

### Конфигурация APScheduler

APScheduler работает как **`AsyncIOScheduler`** в том же event loop, что и бот. Зарегистрирована одна интервальная задача:

| Задача | Интервал | Назначение |
|---|---|---|
| `process_due_reminders` | Каждые N секунд (по умолчанию: 30) | Запрос и отправка наступивших напоминаний |

**Хранилище задач**: Стандартное хранилище в памяти. Расписание задач статично (одна фиксированная интервальная задача), поэтому персистентность не нужна. Фактические данные напоминаний хранятся в PostgreSQL — APScheduler лишь обеспечивает периодический запуск.

### Алгоритм обработки напоминаний

На каждом тике:

```
1. SELECT * FROM scheduled_reminders
   WHERE sent = FALSE AND remind_at <= NOW()
   ORDER BY remind_at ASC
   LIMIT 50

2. Для каждого напоминания:
   a. Загрузить связанное событие (название, дата, время, название пространства).
   b. Отправить Telegram-сообщение пользователю.
   c. Установить sent = TRUE независимо от успеха отправки.
   d. Залогировать ошибки отправки.
```

**Проектные решения**:

- **LIMIT 50**: Предотвращает обработку неограниченного числа напоминаний за один тик. Если наступивших больше 50, следующий тик (через 30 секунд) подберёт остаток.
- **Пометка как отправлено при ошибке**: Напоминание, которое не удалось отправить (пользователь заблокировал бота), всё равно помечается как отправленное. Альтернатива — бесконечные повторы — создала бы постоянно растущую очередь неотправляемых напоминаний. Ошибки логируются для отладки.
- **Нет распределённой блокировки**: Однопроцессная архитектура исключает риск дублирования обработки.

### Гарантия своевременности напоминаний

Напоминания гарантированно отправляются **в пределах одного интервала проверки** (по умолчанию: 30 секунд) от запланированного времени. Это не real-time система — напоминание, запланированное на 18:45:00, может быть доставлено в 18:45:28. Это приемлемо для календарных напоминаний.

## 5. Telegram Mini App (веб-интерфейс)

### Архитектура

Mini App — это **aiohttp web-сервер**, работающий в том же asyncio event loop, что и бот. Он обслуживает:

- **REST API** (`/api/*`) — эндпоинты для работы с пространствами, событиями и настройками пользователя.
- **Статические файлы** — собранный React SPA из `mini_app/frontend/dist/`.
- **SPA-фолбэк** — все запросы, не попавшие в API или статику, отдают `index.html`.

### Аутентификация

Все API-запросы требуют заголовок `Authorization: tma <initData>`, где `initData` — данные, переданные Telegram WebApp при открытии. Сервер:

1. Разбирает `initData` (URL-encoded параметры).
2. Проверяет HMAC-SHA256 подпись с использованием токена бота.
3. Проверяет `auth_date` — не старше 1 часа.
4. Извлекает `user_id` из подписанных данных.

Невалидная или просроченная подпись — HTTP 401.

### API-эндпоинты

| Метод | Путь | Описание | Доступ |
|---|---|---|---|
| `GET` | `/api/spaces` | Список пространств пользователя | Участник |
| `GET` | `/api/spaces/{space_id}` | Детали пространства + участники | Участник |
| `DELETE` | `/api/spaces/{space_id}` | Удалить пространство | Админ |
| `DELETE` | `/api/spaces/{space_id}/members/{user_id}` | Удалить участника | Админ |
| `GET` | `/api/spaces/{space_id}/events` | Ближайшие события пространства | Участник |
| `GET` | `/api/events/{event_id}` | Детали события | Участник |
| `PUT` | `/api/events/{event_id}` | Редактировать событие | Создатель |
| `DELETE` | `/api/events/{event_id}` | Удалить событие | Создатель |
| `GET` | `/api/user/reminders` | Настройки напоминаний | Участник |
| `PUT` | `/api/user/reminders` | Обновить настройки напоминаний | Участник |

### Middleware-стек Mini App

1. **`error_middleware`** — перехватывает исключения, возвращает `{"error": "..."}` с соответствующим HTTP-статусом.
2. **`auth_middleware`** — валидация `initData`, извлечение `user_id`.
3. **`db_session_middleware`** — создание async-сессии, коммит/откат.

### Frontend

React SPA с кастомной дизайн-системой (без внешних UI-библиотек вроде `@telegram-apps/telegram-ui`). Маршрутизация:

| Маршрут | Экран |
|---|---|
| `/` | Список пространств |
| `/spaces/:id` | Детали пространства (участники, события, управление) |
| `/events/:id` | Детали события (редактирование, удаление) |
| `/settings/reminders` | Настройки напоминаний |

Кнопка Telegram BackButton синхронизируется с навигацией React Router.

## 6. Конвенция callback data

Callback data inline-кнопок следует структурированному формату для маршрутизации:

```
{действие}:{id_сущности}:{payload}
```

Примеры:

| Callback Data | Значение |
|---|---|
| `event_confirm:{uuid}` | Подтвердить публикацию события |
| `event_cancel:{uuid}` | Отменить создание события |
| `event_edit:{uuid}` | Перейти к редактированию события |
| `space_select:{uuid}` | Выбрать пространство для события |

Обработчики callback парсят префикс для определения действия и суффикс для контекста. Неизвестные префиксы молча игнорируются.

## 7. Ограничения безопасности

- **API-ключи** (OpenRouter, Nexara, Telegram-токен) хранятся исключительно в файле `.env` и загружаются через `pydantic-settings`. Они не должны появляться в логах, сообщениях об ошибках или системе контроля версий.
- **Invite-коды** генерируются через `secrets.token_urlsafe(6)` — криптографически случайные, 8 символов. Они не являются секретными как таковые (передаются открыто по ссылке), но должны быть неугадываемыми.
- **Контроль доступа** реализован через `AccessControlMiddleware` (invite-only): доступ имеют только владелец бота и участники пространств. Fail-closed при ошибках.
- **Административные операции** (удаление участника, удаление пространства) проверяют роль запрашивающего пользователя в `user_spaces` перед выполнением. Проверка ролей происходит на уровне API-эндпоинта.
- **Аутентификация Mini App** — HMAC-SHA256 валидация `initData` от Telegram WebApp. Срок действия подписи — 1 час. Токен бота не передаётся клиенту.
- **SQL-инъекции** предотвращаются параметризованными запросами SQLAlchemy. Сырые SQL-строки не используются.
- **Валидация ввода**: Ответы LLM валидируются через Pydantic-модель перед использованием. Невалидный или вредоносный вывод LLM не может породить некорректные записи в базе данных.

**Правила слоёв**:

- **Handlers** зависят от **services**. Обработчики содержат Telegram-специфичную логику (форматирование сообщений, построение клавиатур, переходы FSM). Они не содержат бизнес-логики или прямых запросов к БД.
- **Services** зависят от **db/models** и клиентов внешних API. Сервисы содержат бизнес-логику (рабочий процесс создания события, расчёт напоминаний, управление пространствами). Они тестируемы без Telegram.
- **db/models** — чистый слой данных. Никакой бизнес-логики, никаких импортов из других модулей приложения.
- **Клиенты внешних API** (`llm_parser.py`, `speech_to_text.py`) — изолированные сервисы, которые можно заменить или замокать независимо.

## 9. Границы масштабируемости

Это личный бот. Архитектура оптимизирована для **простоты и удобства эксплуатации**, а не для масштаба.

| Измерение | Граница | Обоснование |
|---|---|---|
| Одновременные пользователи | ~50 | Один event loop, один пул соединений к БД |
| Пространства | ~20 | Нет пагинации в команде `/spaces` |
| События на пространство | ~500 | Запросы `/events` просты и проиндексированы |
| Пропускная способность напоминаний | ~100/мин | 50 за тик × 2 тика/мин |
| Задержка API (LLM) | 1–5 секунд | Время ответа OpenRouter; пользователь видит индикатор набора |
| Задержка API (STT) | 2–10 секунд | Время обработки Nexara; зависит от длины аудио |

При достижении любой из этих границ система деградирует плавно (замедление ответов), а не аварийно.
